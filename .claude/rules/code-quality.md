# 実装品質基準

> v2.1.0 · Source: `claudeknowledge` repo → `rules/code-quality.md`。編集はリポジトリ側で行い、`rules/install.sh` で反映する。

`/quality-audit` が事後に検出する問題を、実装段階でゼロにするための基準。
「書いた後に直す」ではなく「最初から正しく書く」。
役割分担: FABLE (`fable-core.md`) が「進め方の規律」、`design-quality.md` が「良い設計・仕様書の基準」
(設計段階での監査先取り)、このファイルが「良いコードの基準」—
FABLE の L (増分実装)・E (検証・セルフレビュー) はこのファイルを基準として適用する。

## 実装前 (毎回必ず)

1. 変更箇所の既存コードを読み、命名・パターン・エラー処理のスタイルを把握する
2. プロジェクトの「正」を確認する: CLAUDE.md / .claude/rules/ / lint・formatter 設定。
   統一の基準は常に「プロジェクトの多数派スタイル + 明文化された規約」— 自分の好みを持ち込まない
3. 同等の処理が既にないか検索する (`grep` / 既存 utils / lib / shared)。あれば再実装せずそれを使う

## コーディング基準

### ハードコーディング禁止 (config)

設定化すべきもの:
- URL・ホスト名・ポート・ファイルパス・バケット名 → 環境変数 / config
- タイムアウト・リトライ回数・バッチサイズ・閾値・キャッシュ TTL・同時実行数 → config or 名前付き定数
- モデル名・デプロイ名・API バージョン → config
- 同じリテラル値が複数箇所に散在 → 定数に集約

そのままでよいもの:
- 数学/プロトコル定数 (86400, 1024, HTTP ステータスコード)、単位変換 → 名前付き定数で十分
- 0, 1, -1 など自明な値 (index, 増分)
- テストの期待値・フィクスチャ
- 一度しか使わない UI 文言 (i18n は別タスク)

追加規則:
- 既存値を設定化するとき、**既定値は必ず現行値を維持する** (設定化と値の変更を同時にやらない —
  挙動不変の保証が崩れる。より良い値に気づいたら別途提案する)
- 環境変数を新設したら、プロジェクトにある**全ての環境定義に揃えて追記する**:
  `.env.example`、docker-compose、IaC (tfvars 等)、k8s manifest、CI の env。
  一部にだけ入れるのが環境差分事故の最頻出パターン

### 重複排除 (DRY)

- 既存の共通 util がある処理を再実装しない — **書く前に検索する**
- 3箇所以上の同一ロジックは共通関数に抽出 (rule of three)
- 2箇所でも、バリデーション・変換・計算ロジックは共通化する
- 共通化先はプロジェクト慣例 (utils/, lib/, shared/, hooks/) に従う
- 偶然似ているだけでドメイン上別概念のものは統合しない (「同時に変更されるべきか?」で判断)

### 命名・リネーム

- 新しい名前はプロジェクトの既存語彙に合わせる (`fetchX` / `getX` / `loadX` の混在を新たに作らない)
- 文字列として外部と契約している名前 (API レスポンスキー・設定キー・DB カラム・イベント名・ルート名)
  は**リネームしない** — 外部契約であり、変えるなら影響調査つきの独立タスク

### エラーハンドリング

- 空の catch / except:pass 禁止 — 最低限ログを出す
- 同種のエラーは同じパターンで処理する (throw vs return None を混在させない)
- リソースは必ず閉じる (context manager / finally / using / defer)
- ユーザー向けエラーに内部情報を含めない (スタックトレース、SQL、内部パス)
- Promise / async は必ずエラーハンドリングする (unhandled rejection を放置しない)

### 型安全

- TypeScript: `any` 禁止、`as` キャスト最小限、`!` non-null assertion 最小限、
  strict で回避できるなら `@ts-ignore` は使わない
- Python: 公開関数には型ヒントをつける、`# type: ignore` は理由コメント付きでのみ使用、
  `Any` は最後の手段
- optional な値は non-null 前提で使わない — 適切にガードする

### デッドコード回避

- 使わないコードを書かない・残さない
- コメントアウトしたコードは削除する (git が履歴を保持している)
- 未使用 import / 変数 / 引数を残さない
- 将来のために「念のため」残すコードは書かない

### セキュリティ

- インジェクション対策: SQL は必ずパラメータ化 (文字列結合/f-string での組み立て禁止)。
  シェルへ外部入力を渡さない (shell=True + 外部入力の組み合わせ禁止)。
  外部入力をパス結合・fetch 先 URL・テンプレート・リダイレクト先に使うときは必ず検証する
  (パストラバーサル / SSRF / SSTI / オープンリダイレクト)
- 危険なシンクに外部入力を流さない: `eval` / `Function` コンストラクタ / `pickle.loads` /
  `yaml.load` (Loader 未指定) / 外部エンティティ解決が有効な XML パーサ (XXE)
- 認証・認可: エンドポイント新設時は認可チェックを必ず入れる。リソース ID は所有者検証する (IDOR 防止)。
  トークンは署名・有効期限・aud/iss を検証する。リクエストボディをフィールド制限なしで
  モデル/DB 更新に直結しない (マスアサインメント)
- 暗号・乱数: パスワード用途に MD5/SHA1 禁止。秘密用途の乱数は `secrets` / `crypto.randomBytes`
  (`random` / `Math.random` は不可)。固定 IV・固定ソルト禁止
- 入力検証: 外部入力は型・サイズをバリデーションしてから使う。外部入力に当てる正規表現は
  破滅的バックトラッキング (ReDoS) を避ける
- 露出防止: ログにトークン・パスワードを出力しない。CORS `*` と credentials の併用禁止。
  デバッグモードを本番設定で有効にしない

### 秘密情報 (secrets)

- API キー・トークン・パスワードをコード・tracked な設定ファイルに書かない — 環境変数 /
  シークレットマネージャ経由。`.env` は untracked が正しい配置、`.env.example` にはキー名のみ (値なし)
- 既存コードやログで secrets を**発見**したら: 値を会話・レポート・コミットメッセージに転記しない
  (file:line とキー名、先頭数文字 + マスク `sk-ab****` のみ)。その時点でユーザーへ即時報告し、
  ローテーションを推奨する — コードから消しても git 履歴と流出済みの可能性は消えない

### 衛生

- デバッグ用 `print()` / `console.log()` を残さない (意図的なロギングは logger 経由)
- TODO にはチケット番号/参照を添える (放置 TODO を作らない)
- ドキュメントとコードの整合を保つ (README のコマンドが動くか、コメントが実装と一致するか)

### 構造・レイヤリング

- プロジェクトのアーキテクチャ層を守る (API→Service→Repository の順序等)
- 1関数1責務 — 取得・変換・保存・通知を1関数に詰め込まない
- 巨大関数 (80行超) / 巨大ファイル (800行超) は責務分割を検討する
- 抽象化の漏れに注意 — provider/adapter パターンからベンダー固有コードを漏出させない

## 実装後セルフレビュー (コミット前に毎回)

**まずツールを回す** (fail-fast の順): lint (`--fix` 適用) → 型チェック / build → テスト。
グリーンでない状態でコミットしない。機械が捕まえられる問題を人力レビューに持ち込まない。

そのうえで diff を敵対的に精読する (FABLE §5 の観点で):

1. ハードコーディングした値がないか / 新設した環境変数が全ての環境定義に揃っているか
2. 既存 util を再実装していないか / 同じロジックを別の場所にも書いていないか
3. エラーパスが全て処理されているか
4. 型が正確か (`any` / 強制キャスト / 型無視がないか)
5. セキュリティ: 外部入力の経路に検証があるか / 新設エンドポイントに認可があるか / secrets が紛れていないか
6. デバッグ残骸がないか (`print`, `console.log`, コメントアウト)
7. テストが変更をカバーしているか
8. 未使用の import / 変数が残っていないか
