# 設計品質基準

> v1.0.0 · Source: `claudeknowledge` repo → `rules/design-quality.md`。編集はリポジトリ側で行い、`rules/install.sh` で反映する。

実装計画・詳細設計・機能仕様を書く段階の品質基準。目標はひとつ —
**この設計に従って実装すれば、`/quality-audit` の全 10 監査次元を最初からパスするコードしか
生まれない**こと。実装後に監査で直すのではなく、監査で見つかる問題を設計の時点で発生不能にする。

役割分担: FABLE (`fable-core.md`) が「進め方の規律」、`code-quality.md` が「良いコードの基準」、
このファイルが「良い設計・仕様書の基準」— FABLE の F (Frame)・A (Anchor)・B (Branch) は
このファイルを基準として適用する。包括的な設計書の生成は `/design-doc` スキルを使う
(このファイルは日々の設計・計画作成用)。

## 適用範囲

- T2: 書面プラン (FABLE F-phase のコントラクト) — 軽量適用: §セルフレビューの 1〜4 のみ必須
- T3: 設計書 (design doc) — このファイルの全項目
- 「実装計画を複数ファイルに分けて作る」「各機能の仕様を先に決める」は全てこのファイルの対象

## 設計前 (毎回必ず) — FABLE Anchor

1. 対象領域の既存コードを読む: 影響を受けるファイル・その呼び出し元・テスト。
   読んでいないコードを前提にした設計をしない
2. プロジェクトの「正」を確認する: CLAUDE.md / .claude/rules/ / 既存の設計書・ADR・
   過去の DECISION ログ。既に決まっていることを再設計しない
3. 再利用調査: 同等の機能・util・パターンが既にないか検索する。設計の第一候補は常に
   「既存の拡張」であり新設ではない。リポジトリ内に無ければ外部を調査する —
   技術選定 (パッケージレジストリ・公式ドキュメント・実績ある実装 — development-workflow §1)
   に加え、方式・アルゴリズム・ドメイン知識は論文・Web も調査対象 (広い論点は
   deep-research スキルへ)。外部情報には出典 (URL/DOI・バージョン・確認日) を付け、
   出典の格 (査読論文・標準仕様 > 公式ドキュメント > 技術記事) を区別する
4. **記憶の API は仮説 (Iron Rule 3)。** 設計書に書く関数・ライブラリ API・フラグ・設定キーは
   実在確認 (grep / 型シグネチャ / 公式ドキュメント) 済みのものだけ。未確認のものは
   `⚠️ assumed` を明記し、verified の顔で書かない
5. 事実タグを内部管理する: verified / inferred / assumed。assumed の上に一方通行の決定
   (スキーマ・公開 API・依存追加) を積まない — 先に検証する

## 設計書の必須要素

章立ての詳細は design-doc スキルの `references/design-template.md` が正。最低限:

- **GOAL・検証可能な受入基準 (AC)・非目標** — FABLE 01-frame の書式。AC は観察可能な挙動で
  2〜5 項目 + 回帰ガード。非目標なしはスコープクリープの温床
- **代替案 ≥2 と DECISION ログ** — FABLE 03-branch の書式。「ボーリングな標準案」を必ず含め、
  各案をスチールマンする。`DECISION: X over Y because Z` を 1 行で記録
- **データモデル・インターフェース契約** (型付き)
- **エラー・失敗モード設計** / **セキュリティ設計** / **設定値インベントリ** (下表)
- **テスト・検証計画** — 各 AC を検証ラダー (L1〜L4) のどこで確認するか
- **移行・ロールバック** (データ・スキーマ・公開 API に触れる場合は必須)
- **未解決事項 (OPEN)** — 未確認の前提・リスクを隠さず列挙

## quality-audit 10 次元を設計で先取りする

各次元について「設計書に書くべきこと」。これが埋まっていない設計は不完全として扱う。

| # | 監査次元 | 設計時に決めること |
| --- | --- | --- |
| 1 | conventions | 新出の名前は既存語彙に揃える (`fetchX`/`getX` の混在を新たに作らない)。設計書に用語集 (概念 → 名前) を載せ、実装時の命名ゆらぎを発生不能にする |
| 2 | config | **設定値インベントリ**: 新規に必要な設定・環境変数・閾値・URL を全列挙 (名前 / デフォルト / 必須性 / 追記すべき環境定義ファイル一覧 — `.env.example`, compose, IaC, CI)。インベントリにない値のハードコーディングは実装時に露見する |
| 3 | dry | 再利用調査の結論を明記: 使う既存 util / 拡張する既存モジュール / 新設するもの。新設には「既存に無いことを検索で確認した」根拠を付ける |
| 4 | structure | 変更・新設ファイルの配置とレイヤ (API→Service→Repository 等) を明記し、各モジュールの責務を 1 行で定義する。1 関数に取得・変換・保存・通知を混載する設計をしない |
| 5 | dead-code | YAGNI: AC に対応しない「念のため」「将来のため」の要素を設計に入れない。置き換えで不要になる既存コード・設定・env var の**削除リスト**を設計に含める |
| 6 | error-handling | 失敗モード (外部 API 障害・不正入力・タイムアウト・競合・部分失敗) ごとに処理方針 (throw / return / retry / ログレベル / ユーザーへの見せ方) を統一的に決める。「エラーは適切に処理」は設計ではない |
| 7 | types | インターフェース契約は型付きで書く (`any` / 素の dict を受け渡す IF を設計しない)。optional の値の扱い (未設定時の挙動) を決める |
| 8 | deps | 依存追加は FABLE 03-branch の依存チェックリスト (stdlib で足りないか / 30 行で書けないか / 健全性 / 供給網) を通し、結果を DECISION として記録する |
| 9 | security | セキュリティ設計章を書く (脅威モデリング — design-doc スキルの `references/security-design.md` が基準)。外部入力の入口・信頼境界・認可モデル・secrets の流れ |
| 10 | hygiene | secrets の注入経路 (env / シークレットマネージャ) とログに出さない項目を設計で決める。TODO をチケット参照なしで設計に残さない |

## 複数ファイル計画の整合 (計画を分割するとき)

- **1 つの overview (正) + 番号付き詳細ファイル** (`00-overview.md`, `01-…`, `02-…`)。
  用語・ファイルパス・AC 番号・DECISION は overview が single source of truth
- 各ファイルの冒頭に: 前提 (先に読むべきファイル) / このファイルのスコープ / 依存する DECISION
- **同じ事実を 2 ファイルに書かない** — 参照する。重複記述は分割計画の矛盾事故の最頻出源
- 実装順序 (ステップ間の依存) を overview に明記する。各ステップは独立に検証可能な増分
  (FABLE L) として切る — 「全部作ってから動かす」計画にしない
- handoff 前に全ファイルを横断整合検査する (design-doc スキルの `references/review-gates.md`)

## トレーサビリティ

- 全ての AC に: それを実現する設計節 + 検証手段 (どのテスト / どの手動確認)。
  対応する設計のない AC、AC に対応しない設計要素は、どちらも欠陥
- 全ての一方通行の決定に DECISION ログ (選ばれなかった案と理由 1 行)

## 設計後セルフレビュー (handoff / 実装着手前に毎回)

設計書を「この実装を拒否したいレビュアー」として敵対的に読む:

1. AC は全て観察可能な挙動か (実装の言い換えになっていないか)。非目標が書かれているか
2. assumed の事実 (未確認 API・記憶のライブラリ仕様) が verified の顔で紛れていないか
3. 代替案が本物か (strawman でないか)。ボーリング案を検討した形跡があるか
4. 別セッションの Claude (または新規参画者) にこの設計だけを渡して、質問なしで実装に
   着手できるか — できないなら情報が足りない
5. 10 次元対応表が全て埋まっているか — 特に設定値インベントリ・エラー失敗モード・削除リスト
6. 全 AC に検証手段があるか。「テストを書く」ではなく「何を入れたら何が返るか」まで書いたか
7. 複数ファイルなら: 用語・番号・パスが横断で一致しているか。矛盾・重複記述がないか
8. secrets・内部認証情報が設計書に転記されていないか (キー名と用途のみ可)
