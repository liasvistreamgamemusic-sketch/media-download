# 設計レビューゲート — Phase 4 の検査項目

全ゲートを実行し、不合格は修正して該当ゲートを再実行する。結果 (N 項目中 N 合格) は
設計書のメタ情報に記載する。ゲート 1〜5 は機械的検査 (自分で照合)、ゲート 6 はエージェント
による敵対的検査。**ゲートを黙って省略しない** — 省略するなら理由を完了報告に書く (Iron Rule 6)。

## ゲート 1: テンプレート完全性

- design-template.md の全章が存在する (該当なしは「該当なし (理由)」で存在させる)
- 各表の列が欠けていない (設定値インベントリの「追記先環境定義」列は特に欠落しやすい)

## ゲート 2: quality-audit 10 次元突合 (設計版監査)

`rules/design-quality.md` の対応表を 1 行ずつ照合する。各次元の合格条件:

| # | 次元 | 合格条件 |
| --- | --- | --- |
| 1 | conventions | 用語集があり、新出の名前が既存語彙と照合済み (`fetchX`/`getX` 混在を新設していない) |
| 2 | config | 設定値インベントリに、設計中の全ての可変値 (URL・閾値・タイムアウト・モデル名等) が載っている。本文を再走査して表にない値を探す |
| 3 | dry | 再利用調査の結論があり、「新設」には検索根拠がある |
| 4 | structure | 全ての新設/変更ファイルにレイヤと責務 1 行がある。レイヤ順序違反 (API→Repository 直結等) を設計していない |
| 5 | dead-code | 削除リストがある (置き換えなのに削除リストが空なら疑う)。AC に対応しない「念のため」要素がない |
| 6 | error-handling | 失敗モード表があり、同種の失敗に同じ処理方針が割り当てられている (throw と return None の混在を設計していない) |
| 7 | types | IF 契約が型付き。`any` / 素の dict の受け渡しを設計していない。optional の未設定時挙動がある |
| 8 | deps | 新依存ゼロ、または依存チェックリスト通過の DECISION がある |
| 9 | security | security-design.md の必須表 (Quick: 表 1・4 / Full: 全表) が埋まっている。対策が「機構」で書かれている |
| 10 | hygiene | secrets の注入経路とログ禁止リストがある。チケット参照のない TODO が設計に残っていない |

## ゲート 3: トレーサビリティ行列

- 全 AC に: 実現する設計節 + 検証手段 (L1〜L4 のどれで・何を入れたら何が返るか) がある
- 逆向き: AC に対応しない設計要素 (どの AC のためでもない機能) がない — あれば削るか AC を直す
- 新規挙動に「変更がないと落ちるテスト」が指定されている
- 検証コマンドがこのリポジトリに実在する (package scripts / Makefile / CI から確認 — 発明禁止)

## ゲート 4: 事実検証 (assumed 掃討)

設計書が前提にする事実を機械的に実在確認する。**エージェント調査の報告は inferred** —
load-bearing なものはここで verified に昇格させる:

- 本文中の関数名・クラス名・設定キー・ファイルパス・API パスを抽出し、grep で実在確認
- 「既存の X を再利用する」の X が本当にその用途で使えるか (シグネチャを読む)
- 外部ライブラリ API は公式ドキュメントまたは型定義で確認 (記憶で書かれた API は最頻出の欠陥)
- 確認できなかったものは verified の顔から `⚠️ assumed` に降格し OPEN 章へ移す
- 本文に「〜と思われる」「おそらく」が残っていないか (二択違反の検出)

## ゲート 5: 複数ファイル横断整合 (分割構成のときのみ)

- 用語: 同一概念が全ファイルで同じ名前か (用語集と突合)
- AC 番号・DECISION 番号: overview の定義と全参照が一致するか (欠番・重複参照の検出)
- ファイルパス・型名・設定キー: ファイル間で表記が割れていないか
- 重複記述: 同じ事実が 2 ファイルに書かれていないか (あれば overview に昇格)
- 依存順序: overview の実装順序と各詳細ファイルの前提が矛盾しないか

## ゲート 6: エージェント敵対レビュー (並列起動)

- **architect 系エージェント**: 設計の穴 — より単純な代替案の見落とし・可逆性・スケールの前提・
  レイヤ配置の妥当性。「この設計を否決する理由を探せ」と指示する
- **security-reviewer 系エージェント**: 脅威モデルの漏れ — 入口列挙の抜け・シンク到達経路・
  認可表の穴。security-design.md を渡して突合させる
- **別人実装可能性テスト**: 設計書**だけ**を渡したエージェントに「この設計から実装手順を
  復唱し、着手に必要な質問を全て挙げよ」と指示する。質問が出た箇所 = 設計の自己完結性の
  不足として本文を修正する (質問ゼロが合格)

指摘は反証してから採用する (エージェントの指摘にも偽陽性はある) — ただし棄却理由は記録する。

## ゲート 7: secrets / 内部情報スキャン

- 実値らしき文字列 (トークン・パスワード・実在ホストの認証情報) が設計書・fact sheet に
  紛れていないか (キー名と注入経路のみが合格)
- 調査中に tracked な secrets を発見していたら、設計書ではなく完了報告で即時報告
  (file:line + キー名 + マスク形式、ローテーション推奨)
