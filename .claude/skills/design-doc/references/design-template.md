# 設計書テンプレート — 章構成と品質基準

design-doc スキルの成果物の基準線。章の順序変更・省略をしない (該当なしは「該当なし (理由)」)。
成果物は 2 種: **機能仕様書** (何を作るか — 読者はレビュアー/将来の開発者) と
**実装計画書** (どう作るか — 読者は実装するエージェント/開発者)。T2 は実装計画書のみで足りる
ことが多い。両方作る場合、仕様書が「正」で実装計画は仕様書を参照する (重複記述しない)。

## A. 機能仕様書 (feature spec)

| 章 | 内容 | 品質基準 |
| --- | --- | --- |
| 1. 目的・背景 | 解決する問題、なぜ今か | 3〜10 行。手段でなく目的を書く (XY 検査済みの目的) |
| 2. スコープ・非目標 | やること / 明示的にやらないこと | 非目標は必須 — 空なら「なぜ空か」を書く |
| 3. 用語集 | この機能の概念 → 採用する名前 | **既存語彙との照合結果を含む** (audit 次元 1 の先取り)。新出概念のみ。実装時はこの表の名前以外を使わない |
| 4. 受入基準 (AC) | 観察可能な挙動 2〜5 項目 + 回帰ガード | FABLE 01-frame の書式 (`AC1: <入力/操作> → <観察可能な結果>`)。番号は以降全文書で共通の ID |
| 5. 機能仕様 | 画面 / エンドポイント / コマンド単位の挙動 | 正常系と境界 (空・0・重複・上限・権限なし) を入出力で書く。「適切に処理」禁止 |
| 6. データ仕様 | エンティティ・属性・関係・制約 | 型・必須性・一意性まで。既存モデルとの関係を mermaid ER 図で |
| 7. エラー仕様 | 失敗モード → ユーザーに見える結果 | 失敗モード列挙 (外部障害・不正入力・タイムアウト・競合・部分失敗) と、それぞれの HTTP コード / メッセージ / リトライ可否 (audit 次元 6 の先取り) |
| 8. セキュリティ要件 | 認可モデル・データ分類・入力検証方針 | security-design.md のチェックリスト準拠 (詳細設計は実装計画側でよいが、認可の「誰が何をできるか」表はここ) |
| 9. 非機能要件 | 性能目標・可用性・データ量の前提 | **実測または明確な予測がある場合のみ**数値を書く。根拠のない数値目標を発明しない |
| 10. 未解決事項 (OPEN) | ⚠️ assumed の前提・保留した論点 | 本文中の ⚠️ が全てここに集約されていること |

## B. 実装計画書 (implementation plan)

| 章 | 内容 | 品質基準 |
| --- | --- | --- |
| 1. コントラクト | TASK / TIER / AC / NON-GOALS の要約 | FABLE 01-frame §5 の書式。仕様書がある場合は AC を参照 (再定義しない) |
| 2. 現状分析 | Phase 1 (Anchor) の調査結果 | 影響ファイルと呼び出し元 / 従うべき既存パターン / **再利用調査の結論** (使う既存 util・「新設が必要」の検索根拠 — audit 次元 3) / 変更不可の外部契約の列挙。全て file:line 根拠付き |
| 3. 代替案と決定 | Branch の結果 | 一方通行の決定ごとに: 代替案 ≥2 (ボーリング案必須) + 評価 + `DECISION: X over Y because Z`。依存追加はチェックリスト通過を明記 (audit 次元 8) |
| 4. 変更ファイル一覧とレイヤ配置 | 新設/変更/削除するファイルと責務 | 各ファイルの層 (API/Service/Repository 等) と責務 1 行 (audit 次元 4)。**削除リスト** — 置き換えで不要になるコード・設定・env var (audit 次元 5) |
| 5. データモデル・IF 契約 | スキーマ / 関数シグネチャ / API 形状 | **型付き**で書く (`any`・素の dict 禁止 — audit 次元 7)。optional の未設定時挙動を明記。マイグレーションは up/down 両方 |
| 6. エラー・失敗モード設計 | 失敗モード → 処理方針 | モードごとに throw/return/retry/ログレベルを表で統一 (audit 次元 6)。リソースの close 責任者を明記 |
| 7. セキュリティ設計 | 脅威モデリング | security-design.md を埋める (T2=Quick / T3=Full — audit 次元 9)。secrets の注入経路とログ禁止項目 (audit 次元 10) |
| 8. 設定値インベントリ | 新規の設定・env var・閾値の全列挙 | 表: 名前 / 型 / デフォルト / 必須性 / 追記先環境定義ファイル (`.env.example`・compose・IaC・CI) (audit 次元 2)。**この表にない値は実装でハードコードしない**、が契約 |
| 9. 実装ステップ | 増分の列と依存順序 | 各ステップ = 独立に検証可能な増分 (FABLE L)。ステップごとに: 変更内容 / 完了条件 / 検証方法 (L1〜L4 のどれで何を確認)。「全部作ってから動かす」構成は不合格 |
| 10. テスト・検証計画 | AC → テストの対応 | **トレーサビリティ表**: AC ID × 実現する設計節 × 検証手段。新規挙動には「変更がないと落ちるテスト」を指定。このリポジトリの実際の検証コマンド (package scripts / Makefile / CI) を使う — 発明しない |
| 11. 移行・ロールバック | データ・互換性・切り戻し | データ/スキーマ/公開 API に触れる場合は必須。ロールバック手順が「ない」ならそれ自体を DECISION として記録 |
| 12. 未解決事項 (OPEN) | ⚠️ assumed・実装前に潰すべき前提 | 「実装前に検証必須」のものに印を付ける |

## 複数ファイル分割の構成規則

分割するのは T3 級 (機能が複数・章が単体で長大化) のときだけ。規則:

```
docs/design/<機能名>/
  00-overview.md      ← 正 (SSOT): コントラクト・用語集・DECISION 全ログ・
                         トレーサビリティ表・実装順序 (依存グラフ)・ファイルマップ
  01-<サブ機能>.md    ← 詳細: 仕様 or 計画の該当章のみ
  02-<サブ機能>.md
```

- 用語・AC 番号・DECISION・パスは **00-overview だけが定義**し、詳細ファイルは参照する
- 各詳細ファイルの冒頭 3 行: 前提 (先に読むファイル) / スコープ / 依存する DECISION 番号
- 同じ事実を 2 ファイルに書かない。書きたくなったら overview に昇格させて両方から参照する
- 番号は読み順 = 実装順に近づける (完全一致は不要、依存グラフが正)

## 文体規則

- 本文は verified の事実と DECISION だけで構成する。「〜と思われる」「おそらく」禁止 —
  verified か `⚠️ assumed` (OPEN 章行き) の二択
- ソースコードを転載しない。書いてよいのは: 型定義・シグネチャ・スキーマ (DDL)・
  API リクエスト/レスポンス例・実行コマンド・`KEY=値` 形式の設定例・mermaid 図
- 表にできるものは表にする (設定値・エラー・トレーサビリティ・認可) — Phase 4 と
  `/quality-audit` が機械的に突合できる形が正
- 各文書の末尾にメタ情報: 対象コミット / 生成日 / ゲート結果 (N 項目中 N 合格)
