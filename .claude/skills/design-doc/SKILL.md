# design-doc — 実装がそのまま監査をパスする設計書を作る

対象機能・要望・チケットを入力に、**機能仕様書** (何を作るか) と**実装計画書** (どう作るか) を
生成する。品質の定義は 2 つ:

1. **この設計だけを渡された別セッションの Claude が、質問なしで実装に着手できる** (自己完結性)
2. **この設計に従って実装したコードは `/quality-audit` の全 10 監査次元を最初からパスする**
   (監査の先取り — 基準は `~/.claude/rules/design-quality.md`)

このスキルは FABLE の F・A・B を設計フェーズの手順として実行するものであり、
別の方法論を持ち込まない。実装 (L・E) はスコープ外 — 設計書の納品で終わり、実装は通常フローへ。

## 絶対原則

1. **読んだ事実だけを設計の前提にする (FABLE Anchor / Iron Rule 3)。** 設計書に書く既存コードの
   挙動・関数・ライブラリ API・設定キーは、このセッションで実在確認したものだけ。
   未確認の前提は `⚠️ assumed` を付けて OPEN 章に集約する — verified の顔で書くのは禁止。
2. **代替案なしに一方通行の決定をしない (FABLE Branch)。** スキーマ・公開 API 形状・依存追加・
   アーキテクチャの継ぎ目は、本物の代替案 ≥2 (ボーリング案を必ず含む) を比較し
   `DECISION: X over Y because Z` で記録してから決める。strawman は代替案ではない。
3. **quality-audit 10 次元を設計で先取りする。** `design-quality.md` の対応表の全項目
   (用語集・設定値インベントリ・再利用調査の結論・レイヤ配置・削除リスト・失敗モード設計・
   型付き IF 契約・依存 DECISION・セキュリティ設計・secrets の流れ) が埋まっていない設計書は
   未完成として納品しない。
4. **テンプレートの章を崩さない。** [references/design-template.md](references/design-template.md)
   の章立てが品質の基準線。該当がない章は省略せず「該当なし (理由)」と 1 行書く。
5. **トレーサビリティ。** 全 AC に実現する設計節と検証手段 (検証ラダー L1〜L4 のどこで確認するか)
   を対応付ける。対応のない AC・AC のない設計要素はどちらも欠陥。
6. **secrets を転記しない。** 調査中に実値が見えても設計書・中間成果物に書かない
   (キー名と注入経路のみ)。tracked な secrets を発見したら即時ユーザーに報告する (マスク形式)。

## 引数の解釈

- **対象**: 設計する機能・要望・チケット番号。Redmine チケット番号なら内容を取得して入力にする。
- **出力先**: 指定があれば従う。なければ Phase 0 で確認 (デフォルト提案: `docs/design/<機能名>/`)。
- **更新モード**: 既存の設計書があり「更新して」「仕様変更を反映して」と指示された場合は、
  影響する章だけ書き直し、DECISION ログは上書きせず追記する (覆した決定は
  `DECISION (reversed): ...` で履歴を残す)。トレーサビリティ表と整合検査は必ず再実行する。

## 全体フロー

```
Phase 0  Frame — 設計種別・tier・出力の確認 (AskUserQuestion 1 回) + AC 起草
Phase 1  Anchor — マルチエージェント現状調査 (影響範囲・既存パターン・再利用・制約)
Phase 2  Branch — 代替案生成と決定 (DECISION ログ)
Phase 3  執筆 (テンプレート章構成に沿って)
Phase 4  敵対的レビューゲート (トレーサビリティ・10 次元突合・横断整合・事実検証)
Phase 5  納品 & 実装フェーズへの引き継ぎ
```

## Phase 0: Frame — AskUserQuestion で 1 回だけ質問 (最大3問)

聞く前に対象を 1〜2 分下見して (関連コード・既存 docs/design/・チケット) 推奨値を作る。
FABLE のティア判定を行い、以降の深さを決める (判定は宣言せず挙動で示す):

- **T2** (単機能・影響範囲が閉じている) → 実装計画書 1 ファイル。軽量適用
- **T3** (アーキテクチャ・スキーマ/データ移行・認証認可・公開 API・複数機能) → 仕様書 + 実装計画
  の複数ファイル構成。全ゲート適用。auth / payments / migration / 公開 API はエスカレータ (+1)

| # | 質問 | 選択肢 (先頭=推奨) |
| --- | --- | --- |
| 1 | 成果物 | ティアから推奨: 「実装計画書のみ (推奨)」 / 「機能仕様書 + 実装計画書」 / 「機能仕様書のみ (実装は未定)」 |
| 2 | 構成 | 規模から推奨: 「単一ファイル (推奨)」 / 「複数ファイル分割 (00-overview + 機能別)」 |
| 3 | 出力先 | 「docs/design/<名前>/ (推奨)」 / 既存 docs 構成に合流 / その他 |

質問と同時にこの時点の理解で **GOAL / AC 案 / 非目標案** を提示し、認識ずれを早期に潰す
(AC の書式は FABLE 01-frame: 観察可能な挙動 2〜5 項目 + 回帰ガード + NON-GOALS)。
要望が手段レベルで具体的すぎる場合は XY 検査 — 目的を確認してから設計する。

## Phase 1: Anchor — マルチエージェント現状調査

設計の全前提をコードに接地させる。観点ごとにエージェントを並列起動する
(Workflow ツールが使えればそれで、なければ Agent ツールの並列起動。読み取り専用):

- **影響範囲**: 変更が必要なファイル・その呼び出し元・既存テスト。外部契約
  (API レスポンスキー・設定キー・DB カラム・イベント名 — リネーム不可のもの) の列挙
- **既存パターン**: この種の機能をこのリポジトリはどう作っているか (レイヤ構成・命名語彙・
  エラー処理様式・テスト様式)。設計はこの多数派スタイルに従う
- **再利用可能物**: 使える既存 util / サービス / 型。「新設が必要」の結論には検索した証拠を付ける
- **制約と先例**: CLAUDE.md / .claude/rules/ / 既存設計書・ADR・過去の DECISION。
  設定値の現状 (既存 env var と環境定義ファイルの一覧 — 設定値インベントリの土台)
- **外部調査** (リポジトリ外の知識が設計の入力になるときのみ起動)。2 系統:
  - *技術選定調査*: パッケージレジストリ (npm / PyPI / crates.io)・公式ドキュメント
    (Context7 / WebFetch)・GitHub code search で、候補ライブラリの比較材料・外部 API の仕様・
    実績ある実装パターンを調べる (development-workflow §1 — 実績ある実装の採用 > 新規実装)
  - *手法・文献調査*: 新機能の方式・アルゴリズム・ドメイン知識を、論文 (arXiv / Google
    Scholar 等) と Web (標準仕様・公式ブログ・技術記事) から WebSearch / WebFetch で調べる。
    論点が広い・結論が設計を左右する場合は **deep-research スキルに委譲**する
    (多ソース突合 + 出典付きレポートが返る)

  外部調査の共通規律: 全ての事実に**出典 (URL/DOI・バージョン・確認日) を付ける** — 記憶で
  書いた外部知識はゲート 4 で落ちる。出典の格を区別する (査読論文・標準仕様 > 公式ドキュメント >
  技術記事) — 設計を左右する主張は独立 2 ソース以上で確認する。論文の性能数値はこの環境で
  再現するまで inferred — 非機能要件の根拠に使うときは「文献値」と明記する。確認できなかった
  事項は ⚠️ assumed。調査結果は比較表にして Phase 2 の代替案候補として渡す

全ての発見に file:line の根拠を付け、fact sheet としてワークスペース
`<repo>/.design-doc/<機能名>/` に保存する (`.git/info/exclude` に追加してリポジトリを汚さない)。
**エージェント報告は inferred** — 設計の土台になる load-bearing な事実 (外部契約・既存 API の
シグネチャ・env var の現状) は Phase 4 までに必ずスポットチェックで verified に昇格させる。

## Phase 2: Branch — 代替案と決定

Phase 1 の事実を入力に、一方通行の決定 (Branch トリガー: スキーマ / 公開 API 形状 / 依存追加 /
アーキテクチャの継ぎ目 / セキュリティ関連フロー / 戻すのに 1 時間超かかるもの) を列挙し、
それぞれに FABLE 03-branch の 3 レンズで本物の代替案を作る:

- **ボーリング案** (このチームの平均的シニアがやる方法・既存コードベースとの最大整合)
- **ミニマル案** (AC を満たす最小 diff。「作らない — 既存 X を設定/再利用」を含む)
- **ロバスト案** (実際に予測される失敗モード・スケールに対応。仮想的リスクには作らない)

評価順序: 正しさ (AC 全充足) > 単純さ > 可逆性 > 一貫性 > 性能 (実測/明確な予測がある時のみ)。
依存追加は 03-branch の依存チェックリストを通す。結果は全て `DECISION: X over Y because Z`。

**ユーザーに聞くもの** (それ以外は推奨を選んで DECISION に理由を書き進む):
製品/UX として見え方が変わるもの・コスト (インフラ/有料 API)・セキュリティ姿勢のトレードオフ・
両案が拮抗し可逆コストが高いもの。AskUserQuestion 1 回に束ね、推奨を先頭にする。

## Phase 3: 執筆

[references/design-template.md](references/design-template.md) の章構成・品質基準に**厳密に**沿う。
複数ファイル構成なら同ファイルの分割規則 (00-overview が正・重複記述禁止・冒頭に前提) に従う。

- セキュリティ設計章は [references/security-design.md](references/security-design.md) の
  チェックリストを埋める形で書く (T2 は Quick セット、T3 は Full セット)
- 設定値インベントリ・削除リスト・トレーサビリティ表は「機械的に照合できる形式」(表) で書く —
  Phase 4 のゲートと `/quality-audit` の突合がそのまま通る形にするため
- 図が伝達に効く箇所 (アーキテクチャ・ER・シーケンス) は mermaid で書く
- 事実タグ: 本文は verified の事実と DECISION だけで構成し、assumed は `⚠️` 付きで OPEN 章へ

## Phase 4: 敵対的レビューゲート

[references/review-gates.md](references/review-gates.md) の全ゲートを実行する。要点:

1. **テンプレート完全性** — 全章存在 (該当なしは理由付き)
2. **10 次元突合** — design-quality.md 対応表を 1 行ずつ照合 (設計版 quality-audit)
3. **トレーサビリティ行列** — AC × 設計節 × 検証手段に穴がないか
4. **事実検証** — 設計書中の関数・API・パス・設定キーを grep で実在確認 (assumed 掃討)
5. **横断整合** (複数ファイル時) — 用語・AC 番号・パス・数値の全ファイル一致
6. **エージェント敵対レビュー** — architect 系 (設計の穴・単純化余地) と security-reviewer 系
   (脅威モデルの漏れ) を並列起動。さらに**別人実装可能性テスト**: 設計書だけを渡した
   エージェントに実装手順を復唱させ、質問が出た箇所 = 設計の不足として修正
7. **secrets / 内部情報スキャン**

不合格は修正して該当ゲートを再実行。ゲート結果 (N 項目中 N 合格) は設計書のメタ情報に記載する。

## Phase 5: 納品 & 実装フェーズへの引き継ぎ

1. 承認された出力先に書き出す。コミットはユーザーの指示があれば行う (git-commit スキルの流儀)。
2. 設計書末尾 (または overview) に **FABLE コントラクトブロック** (01-frame §5 の書式:
   TASK / TIER / AC / NON-GOALS / DECISIONS / STATE / NEXT / OPEN) を含める —
   実装セッションが compaction 後もこれ 1 つで再接地できるようにする。
3. 完了報告 (日本語・結論から): 生成物のパスと構成 / ゲート結果 / DECISION 一覧 /
   ⚠️ OPEN (未確認事項) の件数と、実装前に潰すべきものの指名 / 推奨する実装の進め方
   (増分ステップの先頭と、各ステップの検証方法)。
4. ワークスペース (`.design-doc/`) は残すか消すかを確認する (差分更新の基準に使える)。

## してはいけないこと

- 読んでいないコード・未確認の API を前提に設計する (assumed を verified の顔で書く)
- 代替案なしでスキーマ・公開 API・依存追加を決める / strawman を並べて既定案を通す
- 設定値インベントリ・削除リスト・失敗モード設計を「実装時に決める」で空にする
- AC を実装の言い換えで書く (「関数を追加する」は AC ではない)
- テンプレートの章を黙って省略する / ゲート不合格のまま納品する
- 設計のついでに実装を始める (実装は納品後、通常フローの L・E で行う)
- secrets の実値・内部ホストの認証情報を設計書に書く
- 既存の外部契約 (API キー名・DB カラム・イベント名) のリネームを、影響調査なしに設計へ含める
