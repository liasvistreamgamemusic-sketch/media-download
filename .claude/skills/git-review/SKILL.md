# git-review — ボタンで完結するレビュー対応

## 手順

### 0. ホスト判定と対象一覧

```bash
git remote get-url origin          # github.com → gh / それ以外の GitLab → glab
# GitHub:
gh pr list --limit 4               # 自分に関係するオープン PR
# GitLab:
glab mr list                       # オープン MR
```

- 現在ブランチに紐づく PR/MR があればそれを最有力候補にする
- 0 件なら「対応対象がない」ことを報告して終了 (質問しない)

### 1. コメント取得 (質問の前に必ず)

```bash
gh pr view <番号> --comments       # GitHub
glab mr view <番号> --comments     # GitLab
```

- 指摘を読み、**件数・要旨・対応可否の見立て**を作り、質問の前に本文で一覧表示する
  (指摘 → 見立ての短い箇条書き)。選択肢の説明文には件数などの要点だけを載せる —
  指摘内容が見えないまま対応方針を選ばせない

### 2. AskUserQuestion で 1 回だけ質問

| # | 質問 | 選択肢 (先頭=推奨) | 備考 |
| --- | --- | --- | --- |
| 1 | 対象 | 現在ブランチの PR/MR を先頭に「(推奨)」+ 他のオープン分 (最大4) | 番号+タイトルをラベルに。それ以外は「その他」に番号入力 |
| 2 | 対応方針 | 「全指摘に対応して修正コミット (推奨)」 / 「要約と対応方針の提案だけ」 / 「一部だけ対応」 | 「一部だけ」は「その他」に対象を指定してもらう |

- 対象が 1 件しかなく文脈からも明らかなら質問 1 は省き、方針だけ聞く

### 3. 実行

- 修正は通常の開発規律で行う (FABLE / code-quality 準拠): 指摘箇所を読み、直し、テストを回す
- 指摘が正しくないと考える場合は無視せず、根拠を添えて反論コメント案を提示する
  (投稿はユーザー確認後。投稿するコメントは日本語で書く)
- コミットメッセージは `fix: レビュー指摘対応 (<要旨>)` 等、規約どおり
- 同じブランチに push すれば PR/MR は自動更新される

```bash
git push
```

### 4. 報告 (日本語・結論から)

- どのコメントにどう対応したかの対応表 (指摘 → 対応) を短く
- 未対応・反論したものは理由付きで明示。テスト結果も添える
