# 説明書テンプレート — 章構成・品質基準・文体規則

このテンプレートが説明書の品質の基準線。章の順序・見出しを変えず、該当のない章は
「該当なし (理由を 1 行)」と書いて残す。分割出力の場合は 1 章 = 1 ファイルにし、
目次ファイル (README.md / index.md) から全章へリンクする。

## 文体規則 (全章共通)

- **1 文 1 事実。** 接続詞で事実を連結しない。読者が引用しやすい単位で書く。
- **現在形・断定で書く。**「〜します」「〜です」。「〜と思われる」「おそらく」「たぶん」は禁止 —
  確認できていないなら文末に `⚠️ 未確認` を付け、付録 13 章に登録する。
- **手順は番号付きリスト + 各ステップに期待結果。**「実行する」で終わらせず
  「実行すると○○と表示される」まで書く (読者が成功/失敗を判定できるように)。
- **列挙できるものは表にする。** 環境変数・エンドポイント・コマンド・エラー一覧は必ず表。
  表の説明 (なぜそうなっているか・注意点) は表の前後の本文に書き、セルに詰め込まない。
- **用語は用語集 (12 章) に揃える。** 同じ概念を別の語で呼ばない (「ジョブ」と「タスク」の混在等)。
  コード内の識別子名を使う場合は初出時に日本語の説明を添える。
- **読者は開発者。** 一般的な技術用語 (トランザクション・冪等・キュー等) は説明なしで使ってよい。
  ただしプロジェクト固有の概念・略語・ドメイン用語は初出で説明し、用語集に載せる。
- **詳細さを恐れない。** 開発者向け文書では「省いて短くする」より「詳細で正確」が正義。
  処理の流れは分岐条件・トランザクション境界・副作用 (保存・通知・外部呼び出し) まで書く。
  ただし詳細=コード転載ではない — 「何が・いつ・どういう条件で起きるか」を日本語で書く。
- **コード非転載** (SKILL.md 原則 2): 挙動は文章で。載せてよいのはコマンド・設定例 (`KEY=値`)・
  ディレクトリツリー・mermaid 図・API リクエスト/レスポンス例・エラーメッセージ文字列のみ。

## 0. ドキュメント情報

- 対象リポジトリ / 対象範囲 (scope 限定時はその旨)
- 対象コミット (`git rev-parse --short HEAD`) と生成日
- カバレッジ実績: 対象 N ファイル / 抽出済み N / 除外 M (理由別)、環境変数 K 件中 K 件掲載
- 凡例: `⚠️ 未確認` = コードから確認できなかった記述 (一覧は 13 章)
- 更新方法: このスキルでの差分更新手順を 1〜2 行

## 1. 概要

新規参画の開発者が 3 分で「これが何か」を掴める章。

- **目的**: 何のためのシステムか。解決する課題を 2〜3 文で
- **主要機能一覧**: 表 (機能名 / 一言説明 / 詳細は 5 章の該当節へリンク)
- **利用者**: 誰がどう使うか (エンドユーザー / 管理者 / 他システム)
- **技術スタック**: 言語・フレームワーク・主要ミドルウェアを 1 行ずつ (バージョンは manifest から)

品質基準: この章だけ読んで、プロジェクトを 1 段落で他人に説明できること。

## 2. システム構成

- **アーキテクチャ図** (mermaid): コンポーネント・データストア・外部サービスと、その間の通信
- **コンポーネント一覧**: 表 (名前 / 責務 / 配置ディレクトリ / 依存先)
- **外部サービス依存**: 表 (サービス名 / 用途 / 障害時の影響 / 関連する環境変数 → 4 章リンク)
- **主要データフロー**: 代表的なリクエスト/処理が入口から出口までどう流れるか (文章 or mermaid シーケンス図)

品質基準: 障害調査で「どこを疑うか」の当たりがこの章だけで付けられること。

## 3. セットアップ

新規参画の開発者がコードを読まずに、ゼロから起動・動作確認まで完走できる章。

- **前提ソフトウェア**: 表 (名前 / 必要バージョン / 確認コマンド)。バージョンの根拠は
  manifest (engines, requires-python 等) / Dockerfile / CI 設定
- **手順**: clone → 依存インストール → 設定 (4 章の必須変数への参照) → 初期データ/マイグレーション → 起動。
  各ステップに期待結果を書く
- **動作確認**: 起動後に何をどう叩けば「動いている」と言えるか (URL・コマンドと期待レスポンス)
- **複数の起動方法** (ローカル / Docker 等) がある場合は推奨を明示し、それぞれ完結した手順にする

品質基準: Phase 4 の机上完走テストに合格すること。手順間に「その値はどこから?」という飛躍がないこと。

## 4. 設定リファレンス

このスキルの品質保証の中核。**環境変数レジストリの全変数を載せる — 間引き禁止。**

### 4.1 環境変数

カテゴリ (アプリ基本 / データベース / 外部サービス / 認証 / 機能フラグ / チューニング…) ごとに表を分ける:

| 変数名 | 必須 | デフォルト | 取りうる値 | 説明 | 変更時の影響 |
| --- | --- | --- | --- | --- | --- |

各列の記入基準:

- **必須**: `必須` (未設定で起動失敗/機能不全) / `任意` (デフォルトで動く) / `条件付き必須` (例: 「○○機能を使う場合のみ」)。
  根拠は読み取りコードの挙動 — バリデーションで落ちるのか、フォールバックがあるのか。
- **デフォルト**: 読み取りコードのフォールバック値をそのまま。デフォルトなし (undefined/None のまま進む) なら `なし`。
- **取りうる値**: コードの分岐・比較・enum・バリデーションから列挙する。
  列挙型なら全値 (`production` / `staging` / `development`)、数値なら型と単位と実用範囲 (「秒。正の整数」)、
  真偽値ならコードが真と解釈する表記 (「`true` / `1` (それ以外は偽)」)、自由形式なら形式と例 (「URL。例: `https://…`」)。
  「文字列」だけで終わらせない — この列が読者の設定ミスを防ぐ。
- **説明**: 何を制御するか 1〜2 文。
- **変更時の影響**: 変えると何が起きるか / 再起動が必要か / 本番で触ってよいか。

secrets 系変数 (キー・トークン・パスワード) は値の例を書かず「取得方法 or 発行元」を説明に書く。

### 4.2 設定ファイル

表 (ファイルパス / 形式 / 役割 / 主要キーの説明)。環境変数と重複する設定は優先順位を明記する。

### 4.3 環境定義の整合

環境変数 × 環境定義ファイル (`.env.example` / docker-compose / CI / IaC) のマトリクス。
欠落 (どこかに載っていない変数) は ❌ で示し、13 章の矛盾リストにも載せる。

品質基準: grep レジストリの変数数と 4.1 の行数が一致すること (0 章に実績を記載)。

## 5. 機能仕様

主要機能ごとに同じ骨格で書く (骨格を揃えることが品質の一定性):

- **目的**: この機能は何のためにあるか
- **トリガー / 入口**: どこから起動するか (画面・API・スケジュール・イベント)
- **入力**: 何を受け取るか・バリデーション条件
- **処理の流れ**: 文章で段階的に (コードは貼らない)。分岐条件は「〜の場合は…」で明示
- **出力 / 結果**: 何が返る・何が保存される・何が通知されるか
- **エラー時の挙動**: 失敗パターンごとに何が起きるか (リトライ? ロールバック? 通知?)
- **関連設定**: この機能に影響する環境変数・設定 (4 章へリンク)

品質基準: 各機能について「期待挙動の仕様」としてテストケースを書ける粒度であり、
改修時に「どの機能・設定・データに影響するか」をこの章と 4 章の相互リンクから判断できること。

## 6. インターフェース仕様

- **API**: エンドポイント一覧表 (メソッド / パス / 認証要否 / 概要)。主要エンドポイントは
  リクエスト/レスポンス例 (JSON) とエラーレスポンス形式を添える
- **CLI / スクリプト**: 表 (コマンド / 引数・オプション / 何をするか / 定義元)
- **画面 / ルート**: 一覧表 (パス / 画面名 / 誰が使うか)
- **Webhook / イベント / 他システム連携**: 受け口と送り先

## 7. データ仕様

- **ER 図** (mermaid): 主要エンティティと関係
- **エンティティ一覧**: 表 (名前 / 役割 / 主要フィールドの意味・制約・取りうる値)。
  status 等の enum カラムは全値と遷移を書く
- **永続化先**: DB / ストレージ / キャッシュの使い分け
- **マイグレーション運用**: 追加・適用・ロールバックの手順とコマンド

## 8. バッチ・ジョブ・非同期処理

表 (ジョブ名 / 起動条件 (スケジュール式は人間語に翻訳: `0 3 * * *` → 「毎日 3:00」) / 処理概要 /
失敗時の挙動・リトライ / 多重起動の可否)。該当なしなら「該当なし」。

## 9. 運用ガイド

- **起動・停止・再起動**: 環境ごとのコマンドと確認方法
- **ログ**: 出力先・形式・レベルの変え方 (関連環境変数 → 4 章)
- **監視**: ヘルスチェックエンドポイント・見るべきメトリクス
- **バックアップ / リストア**: 対象データと手順 (仕組みが無ければ「未整備 ⚠️」と正直に書く)
- **デプロイ**: CI/CD の流れ (トリガー → ビルド → デプロイ先)、手動手順があればそれも

## 10. 開発ガイド

- **リポジトリ構成**: ディレクトリツリー (2〜3 階層) + 各ディレクトリの役割 1 行
- **開発コマンド**: 表 (目的 / コマンド / 定義元)。テスト・lint・format・build・ローカル起動
- **テスト**: 実行方法・テストの分類 (unit/integration/E2E)・書き方の慣例
- **CI/CD**: パイプラインの段階と、落ちたとき開発者が何をすべきか
- **規約**: ブランチ命名・コミット形式・レビュー運用 (CLAUDE.md / rules に定義があればそれを要約しリンク)

## 11. トラブルシューティング

コードのエラー処理・ログ出力から**逆引きで**生成する: 読者が遭遇する症状 → 原因 → 対処。

| 症状 (エラーメッセージ / 挙動) | 原因 | 対処 |
| --- | --- | --- |

- エラーメッセージはコードにある文字列をそのまま載せる (検索で辿り着けるように)
- 設定ミス起因 (env var 未設定・値不正) のものを優先的に網羅する — 最頻出のため
- セットアップ時のつまずき (ポート衝突・バージョン不一致・依存欠落) も含める

## 12. 用語集

表 (用語 / 定義 / コード内での識別子名)。ドメイン用語・プロジェクト固有の略語・
コード内の主要な概念名を載せる。本文はこの用語集の語彙に統一する。

## 13. 付録

- **未確認事項**: 本文の `⚠️ 未確認` 全件のリスト (場所 / 内容 / 確認するには何が必要か)
- **既存ドキュメントとの矛盾**: 既存 README / docs とコードが食い違っていた点 (コードを正として記述した旨)
- **環境定義の欠落**: 4.3 で見つかった ❌ の一覧 (修正は別タスクとして提案)
