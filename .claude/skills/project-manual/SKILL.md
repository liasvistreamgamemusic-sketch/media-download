# project-manual — コードから生成する網羅的な開発者向け説明書

コードベースを唯一の情報源として、プロジェクトの開発者向け説明書 (マニュアル / 仕様書) を生成する。
「コードを読める人の記憶」ではなく「コードそのもの」から事実を抽出し、
**新規参画の開発者が全コードを読まなくても、仕様の理解・環境構築・運用・開発着手までを
独力で完了できる**文書に変換する。詳細さと正確さを落とさないこと — 詳細度の基準は
「この文書と 4 章の設定リファレンスだけで、機能改修のテストケースと影響範囲を挙げられる」。

## 絶対原則

1. **読んだ事実だけを書く (FABLE Anchor)。** 説明書の全ての記述は、このセッションで実際に読んだ
   コード・設定・定義に根拠を持つ。根拠は執筆中 file:line で内部管理する (文書本体には載せない)。
   環境変数・API・コマンド・設定キーの発明は絶対禁止 (Iron Rule 3)。
   コードから確認できなかったことは本文に `⚠️ 未確認` を付け、付録の未確認リストに集約する。
   「〜と思われる」「おそらく」で濁して書くのは禁止 — verified か ⚠️ かの二択。
2. **ソースコードを転載しない。** 関数・クラスの抜粋を貼らず、挙動を文章・表・図で説明する。
   載せてよいのは: 実行コマンド、設定例 (`KEY=値` 形式)、ディレクトリツリー、mermaid 図、
   API のリクエスト/レスポンス例、エラーメッセージ文字列。これらは「読者が使うもの」でありコードではない。
3. **網羅性を機械的に保証する。** 「だいたい書いた」は禁止。
   カバレッジマニフェスト (全ファイル → 抽出済み) と環境変数レジストリ突合
   (grep で検出した変数数 = 文書の表に載った数) で完全性を検証し、実績を文書に記載する。
4. **章構成を崩さない。** [references/manual-template.md](references/manual-template.md) の章立てが
   品質の基準線。該当がない章は省略せず「該当なし (理由)」と 1 行書く —
   「書いていない」と「存在しない」を読者が区別できることが説明書の品質そのものであるため。
5. **デフォルト値・必須性はコードが正。** README や既存ドキュメントの記述は参考情報であり根拠にしない。
   env var のデフォルト値は必ず読み取りコード (フォールバック値) から取る。
   既存ドキュメントとコードが矛盾していたらコードに従って書き、矛盾リストとして付録+完了報告に載せる。
6. **secrets を転載しない。** 実際の値が `.env` や設定から見えても、文書・中間成果物・レポートに
   値を書かない (キー名と用途のみ)。発見した tracked な secrets は即時ユーザーに報告する (マスク形式)。

## 引数の解釈

- **scope**: 対象範囲。デフォルトはリポジトリ全体。「backend だけ」等の指定があれば限定し、
  文書冒頭に対象範囲を明記する。
- **出力先**: 指定があればそれに従う。なければ Phase 0 で確認する。
- **更新モード**: 過去のワークスペース (`.project-manual/`) と生成済みマニュアルが存在し、
  ユーザーが「更新して」「差分反映して」と指示した場合は、前回生成コミット以降の
  `git diff --name-only` の変更ファイルのみ再抽出し、影響のある章だけ書き直す。
  環境変数レジストリと付録の突合・メタ情報 (対象コミット等) は必ず再生成する。

## 全体フロー

```
Phase 0  Frame — 読者・出力・言語の確認 (AskUserQuestion 1 回)
Phase 1  インベントリ作成 (カバレッジマニフェスト)
Phase 2  マルチエージェント事実抽出 (チャンク抽出 → 横断集約 → 検証)
Phase 3  執筆 (テンプレート章構成に沿って)
Phase 4  敵対的セルフレビュー & カバレッジ検査
Phase 5  納品 & 完了報告
```

中間成果物 (fact sheet JSON・レジストリ・進捗) はワークスペース `<repo>/.project-manual/<YYYYMMDD>/` に
保存し、中断しても再開できるようにする。このディレクトリは `.git/info/exclude` に追加して
リポジトリを汚さない (tracked な .gitignore は変更しない)。

## Phase 0: Frame — AskUserQuestion で 1 回だけ質問 (最大3問)

**読者は常に開発者** (新規参画者〜運用担当の開発者)。読者層は聞かない —
専門用語は使ってよいが、プロジェクト固有の概念・略語は初出で説明し用語集に載せる。

聞く前にリポジトリを 1〜2 分下見して (README・ディレクトリ構成・package manifest)、推奨値を作る:

| # | 質問 | 選択肢 (先頭=推奨) |
| --- | --- | --- |
| 1 | 出力形式 | プロジェクト規模から推奨を判定: 小〜中は「単一ファイル `docs/MANUAL.md` (推奨)」 / 大は「章ごとに分割 `docs/manual/` (推奨)」 / もう一方 / 「既存 docs 構成に合流」 |
| 2 | 言語 | 既存ドキュメントの言語に合わせたものを推奨 (このユーザーは通常「日本語 (推奨)」) / 英語 / 両方 |
| 3 | 重点 | 「バランス (推奨)」 / 「運用重視 (設定・トラブルシュート厚め)」 / 「仕様重視 (機能・API・データ厚め)」 |

推測できるものは聞かない — 推測値を推奨ボタンにして確認だけ取る。引数で指定済みの項目は省く。

## Phase 1: インベントリ作成

1. `git ls-files` で全 tracked ファイルを取得し、抽出対象外を除外する:
   lock ファイル、ビルド成果物、`*.min.*`、vendored コード、バイナリ/画像。
   除外はマニフェストに「除外 + 理由」で残す (漏れではなく意図的除外であることを示す)。
   **テスト・CI 設定・IaC・マイグレーションは除外しない** — 仕様 (期待挙動)・運用・データの一次情報源。
2. 対象をチャンクに分割する (ディレクトリ・レイヤー単位、1 チャンク = 20〜40 ファイル or 3,000〜5,000 行目安)。
3. `manifest.json` (全ファイル → チャンク ID) をワークスペースに保存する。

## Phase 2: マルチエージェント事実抽出

抽出する事実の種類・fact sheet スキーマ・エコシステム別の grep パターンは
[references/extraction.md](references/extraction.md) を読み、エージェントのプロンプトに該当部分を含めること。

構成は 3 層。実行手段: Workflow ツールが利用可能ならそれを使う (このスキルの指示による呼び出しは
明示的オプトインに該当する)。なければ Agent ツールの並列起動で同じ構成を組む。

- **Layer A — チャンク抽出** (チャンクごとに 1 エージェント、並列):
  担当チャンクの全ファイルを精読し、**コードではなく事実**を構造化して返す —
  機能 (何がどう動くか)、環境変数 (名前・読み取り箇所・フォールバック値・分岐で使われる値)、
  設定キー、エンドポイント/コマンド/画面、データモデル、ジョブ/キュー、外部サービス依存、
  エラーメッセージとその発生条件。全事実に file:line の根拠を付ける。
- **Layer B — 横断集約** (観点ごとに 1 エージェント、並列。Layer A 完了後):
  ①**環境変数レジストリ**: リポジトリ全体 grep (extraction.md のパターン) で env var を独立に検出し、
  Layer A の申告と突合して完全な一覧を作る。各変数について読み取り箇所を精読し、
  必須性 (未設定時に落ちるか・デフォルトで動くか)・デフォルト値・取りうる値
  (比較/分岐/enum/バリデーションから列挙)・変更時の影響を確定する。
  さらに `.env.example` / compose / CI / IaC の各環境定義に載っているかの整合表を作る。
  ②アーキテクチャとデータフロー (エントリポイントから追う)。③API/CLI/画面のサーフェス全体。
  ④データモデル全体と関係。⑤セットアップ〜起動の動線 (README・Dockerfile・compose・CI を突合)。
- **Layer C — 事実検証** (並列。文書の信頼性を左右する load-bearing な事実が対象):
  環境変数のデフォルト値と必須性 (レジストリの記述をコードで再確認)、セットアップ手順のコマンド
  (定義元 — package.json scripts / Makefile / CI — に実在するか)、API のパスとメソッド、
  デフォルトポート/URL。反証が出たら Layer B の結果を修正する。
  コマンドの実行検証はしない (存在確認まで) — ただし lint / build / test など安全なものは、
  ユーザー環境を変えない範囲で 1 回実行して「動作確認済み」マークを付けてよい。

**待ち時間を作らない:** Layer C の検証はチャンク単位で始められるものから始める。
バリアが必要なのは Layer B (全チャンクの fact sheet が入力) だけ。
抽出エージェントの reasoning effort は下げない — 説明書では読み落とし (偽陰性) が最も高くつく。

## Phase 3: 執筆

[references/manual-template.md](references/manual-template.md) の章構成・各章の品質基準・文体規則に
**厳密に**沿って執筆する。章の順序変更・省略をしない (該当なしは「該当なし」と明記)。

- 執筆は章単位。大規模プロジェクトでは章ごとにエージェント分担してよいが、
  用語集・凡例・トーンを揃えるため、テンプレートと用語集ドラフトを全員に渡す。
- 環境変数リファレンスの表は Layer B レジストリから**機械的に**生成する (執筆時の要約で情報を落とさない)。
- 図は mermaid で書く (アーキテクチャ図・ER 図・主要フローのシーケンス図)。
- 根拠 (file:line) は本文に載せず、ワークスペースの `evidence.json` に章→事実→根拠で保存する
  (次回の差分更新と、レビュー時の検証に使う)。

## Phase 4: 敵対的セルフレビュー & カバレッジ検査

文書全体を「このプロジェクトを知らない新規参画の開発者」として通読し、以下を機械的に検査する:

1. **カバレッジ**: マニフェストの全対象ファイルがいずれかの fact sheet でカバーされているか。
   環境変数レジストリの全変数が設定リファレンスの表に載っているか (数の一致を確認)。
   検出したエンドポイント/コマンド/モデルが全てインターフェース仕様・データ仕様に載っているか。
2. **セットアップ完走テスト (机上)**: セットアップ章の手順だけで、前提 → インストール → 設定 →
   起動 → 動作確認まで、コードを読まずに完走できるか。手順間の飛躍 (「その値どこから?」) がないか。
3. **コード転載ゼロ**: 原則 2 で許可したもの以外のコード片が紛れていないか。
4. **未確認の集約**: 本文の `⚠️ 未確認` が全て付録の未確認リストに載っているか。濁し表現が残っていないか。
5. **secrets ゼロ**: 実値らしき文字列 (トークン・パスワード・実在ホストの認証情報) が紛れていないか。
6. **章の完全性**: テンプレートの全章が存在するか。表の列が欠けていないか。
7. **整合**: 用語が用語集と一致しているか。既存 README 等との矛盾が付録に列挙されているか。

不合格項目は修正して再検査する。検査結果 (対象 N ファイル / 抽出済み N / 除外 M、env var K 件中 K 件掲載)
は文書のドキュメント情報章に記載する。

## Phase 5: 納品 & 完了報告

1. 承認された出力先に書き出す。コミットはユーザーの指示があれば行う (git-commit スキルの流儀に従う)。
2. 完了報告 (日本語・結論から): 生成した文書のパスと章構成 / カバレッジ実績 /
   ⚠️ 未確認事項の件数と一覧 / 既存ドキュメントとの矛盾リスト / 環境定義ファイルの整合で見つかった欠落
   (例: `.env.example` に載っていない変数) / 更新方法 (「次回は差分更新できる」)。
3. ワークスペース (`.project-manual/`) は残すか消すかをユーザーに確認する (差分更新の基準に使えるため)。

## してはいけないこと

- コードから確認していない挙動・値・手順を書く (既存 README からの転記は「未確認」扱い)
- ソースコードの関数・クラスを文書に貼る
- テンプレートの章を黙って省略する / 環境変数の表から「重要でなさそうな変数」を間引く
- デフォルト値を README や記憶から書く (必ず読み取りコードのフォールバック値から)
- secrets の実値を文書・fact sheet・レポートに書く
- 検証していないコマンドに「動作確認済み」と書く (存在確認のみなら定義元を根拠として書く)
- 説明書生成のついでにコードを「修正」する (矛盾や不備は報告に留め、修正は別タスクとして提案)
