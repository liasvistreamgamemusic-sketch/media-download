# カテゴリ別の修正ポリシー

Phase 5 で各カテゴリを修正する際の安全規則。共通原則は SKILL.md の「絶対原則」—
ここでは「このカテゴリをどう直せば挙動を壊さないか」を定める。

## 共通: テストゲート

各カテゴリの修正完了ごとに必ず実行する (fail-fast の順):

1. lint (自動修正可能なものは `--fix` で適用)
2. 型チェック / build
3. テスト一式 (ベースラインで既に落ちていたものは判定から除外)

グリーンになって初めて `git add` → コミット。部分的にしか直っていない状態でのコミットはしない。
テストカバレッジが薄い領域の修正は、変更前後で同じ入出力になることを手元で確認する
(REPL・一時スクリプト・既存テストの一時的な追加実行など、確認手段を自分で作る)。

## 1. 秘密情報の除去 (hygiene / security)

- 発見時点でまずユーザーに即時報告し (マスクした形で)、**ローテーションを推奨する**。
  コードから消しても git 履歴と流出済みの可能性が残るため、削除はローテーションの代替にならない。
- 修正は「環境変数 / シークレットマネージャ参照への置換 + `.env.example` へのキー名 (値なし) 追記」。
  実際の値の移送 (`.env` への書き込み等) はユーザー自身に依頼する — エージェントが値を扱う工程を作らない。
- 置換後、そのキーが必要な全環境定義 (docker-compose, CI variables, IaC) にキー名を揃えて追記する
  (§マジックナンバーの環境変数新設と同じ規則)。
- コミットメッセージ・レポート・diff の説明に値を書かない (キー名と場所のみ)。
- git 履歴からの完全除去 (`git filter-repo` 等) は履歴書き換えで破壊的なため、提案のみで実行しない。

## 2. デッドコード削除

- 削除対象は「Layer C 検証済み + (ツールがあれば) ツールでも未使用判定」のもののみ。
  どちらか一方しか確認できていないものは削除せず「報告のみ」に残す。
- シンボル削除に伴う未使用 import・空ファイル・空ディレクトリも一緒に掃除する
  (lint がそれらを検出するので、lint を掃除の完了確認に使う)。
- public API (パッケージとして公開されているもの・他リポジトリが消費するもの) は削除ではなく
  deprecation を提案する。
- コメントアウトされたコードブロックは削除してよい (git 履歴に残っているため)。
  ただし「なぜ無効化したか」の説明コメントを伴う場合は、その意図が現在も有効か判断できなければ残す。
- 機能フラグの残骸は「フラグが全環境で恒久値になっている」ことを設定ファイル側で確認してから畳む。

## 3. 規約統一

- まず formatter / lint の自動修正 (`ruff check --fix && ruff format`, `eslint --fix`, `prettier --write` 等)
  を対象範囲に一括適用する。手動修正はその後の差分に対してだけ行う (手戻り防止)。
- 命名のゆらぎ統一はシンボルリネームなので、LSP やツールの rename 機能・全参照 Grep で
  呼び出し側も漏れなく更新する。文字列参照 (API レスポンスキー、設定キー、DB カラム名) に
  あたる名前は **外部契約なのでリネームしない** — ローカル変数・内部関数に限定する。
- lint 設定自体の変更 (ルール追加・厳格化) は挙動こそ変えないが影響が大きいため、
  承認時に独立項目として明示されている場合のみ行う。

## 4. マジックナンバー / ハードコーディングの設定化

- **既定値は必ず現行値を維持する。** 設定化と値の変更を同時にやらない (挙動不変の保証が崩れるため)。
  「より良い値」に気づいたら finding として報告だけする。
- 設定機構はプロジェクトの既存のもの (pydantic Settings, 環境変数 + config モジュール,
  フレームワークの config) に従う。新しい設定方式を導入しない。既存機構が本当に無い場合のみ、
  そのエコシステムで最も標準的な最小手段を使い、その旨を報告する。
- 環境変数を新設する場合、プロジェクトに複数の環境定義ファイルがあれば**全てに揃えて追記する**:
  `.env.example` / `.env` 系、IaC (terraform tfvars 等)、docker-compose、k8s manifest、CI の env。
  一部の環境にだけ入れると環境差分事故の原因になる。
- 数学・プロトコル定数は設定化せず、意味の分かる名前付き定数にする
  (`SECONDS_PER_DAY = 86400`)。置き場はそのモジュール内 or プロジェクトの constants 慣例に従う。
- 同じ値が複数箇所に散っている場合は、設定化/定数化と同時に参照を1点に集約する。

## 5. 重複コードの共通化

- 統合前に「この N 箇所は本当に同じセマンティクスか (同時に変更されるべきものか)」を再確認する。
  迷ったら統合しない — 誤った抽象化は重複より高くつく。
- 共通化先はプロジェクトの慣例の場所。新しい util 名はプロジェクトの命名規約に従う。
- 手順: ①共通関数を抽出 (まず1箇所の実装をそのまま移す) → ②テスト実行 → ③残りの呼び出し元を
  1つずつ置換し、都度差分がないか確認 → ④テスト実行。微妙に異なる重複は、差分が bug なのか
  意図なのかを判断し、意図的な差分はパラメータ化する (bug 疑いは挙動変更になるため報告のみ)。
- 共通化対象にテストがない場合、可能なら先に特性テスト (現在の入出力を固定するテスト) を書いてから
  統合する。テストを書くコストが見合わない小物は、置換前後の呼び出し結果を手元で突合する。

## 6. エラーハンドリング・型安全・ロギングの一貫性

- 握りつぶしの解消は挙動変更になりうる (今まで握りつぶされていた例外が伝播する) ため、
  原則 behavior_change=possible として承認時に明示し、承認された場合のみ直す。
  安全に直せるのは「ログ追加のみ」「既に到達不能な catch の整理」など。
- 型の修正 (`any` → 具体型、型ヒント追加) はランタイム挙動を変えないので安全。ただし
  TypeScript で型を厳しくした結果あぶり出される既存バグは、修正せず finding として報告する
  (バグ修正は挙動変更なので別タスク)。コンパイルが通らなくなる場合のみ最小限の対処をする。
- ログレベル・フォーマットの統一はプロジェクトの既存ロガー設定に従う。ログ出力の追加・削除は
  監視・アラートに影響しうるため、エラーパス以外への大量追加はしない。

## 7. 構造改善

- 実施するのは承認時に個別に了承されたもののみ。
- 1つの構造変更 = 移動 + import 更新を1セットで完結させ、中間状態でテストを通す。
- public な import パスが変わる場合は、旧パスに re-export を残して互換を保つことを優先検討する。
- ファイル分割・関数分割は「コードの移動」に徹し、ロジックの書き換えを混ぜない
  (diff レビューで移動と変更を区別できなくなるため)。
- 規模が想定の2倍を超えそうだと分かった時点で中断し、状況を報告して別タスク化を提案する。

## コミット

- 1カテゴリ = 1コミット。メッセージはプロジェクト規約 (形式・言語・チケット参照) に従う。
  規約がなければ Conventional Commits: `refactor: <カテゴリの内容を1行で>`。
- コミット本文には「何を・何件・どの方針で」直したかを書く。レポートの finding ID を列挙してよい。
- 1カテゴリの差分が巨大になる場合 (目安: 50 ファイル超) は、レビュー可能性のために
  サブカテゴリ (例: `refactor: 未使用import削除 (backend)` / `(frontend)`) に割ってよい。
  ただし各コミットが独立してテストグリーンであること。
