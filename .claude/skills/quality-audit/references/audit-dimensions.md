# 監査次元チェックリスト

各次元の「見つけるべきもの」と「flag してはいけないもの (偽陽性)」。
Layer A / B エージェントへのプロンプトには、担当次元の該当セクションを含めること。
重大度 (severity) は high / medium / low、リスク (risk) は safe / low / medium / high、
挙動変更 (behavior_change) は none / possible / yes で分類する。

## 1. conventions — コード規約・スタイル統一

**見つけるもの:**
- プロジェクト規約 (CLAUDE.md / .claude/rules/ / lint 設定) からの逸脱
- 同じ概念に対する命名のゆらぎ (`fetchUser` / `getUser` / `loadUser` が混在等)
- import の順序・グルーピングの不統一
- ファイル/ディレクトリ配置の規約逸脱 (コンポーネントの置き場所、テストの配置)
- コメント言語・docstring スタイルの不統一 (プロジェクト規約がある場合)
- 同種の処理での書き方の分裂 (例: 同じ API 呼び出しで async/await と .then が混在、
  片方は早期 return で片方はネスト)

**flag しないもの:**
- formatter が自動で直せるもの (それは Phase 5 で一括 `--fix` するだけなので個別 finding にしない。
  「formatter 未適用ファイルが N 件」と集約して報告)
- 好みの問題でプロジェクト内に明確な多数派がないもの
- 外部ライブラリの API に合わせざるを得ない命名

**統一の基準:** 常に「プロジェクト内の多数派」または「明文化された規約」。判断材料として
出現数を Grep でカウントし、少数派を多数派に合わせる方向で提案する。

## 2. config — マジックナンバー / ハードコーディング

**見つけるもの (→ 設定化・環境変数化を提案):**
- 環境によって変わりうる値: URL・ホスト名・ポート・ファイルパス・バケット名
- チューニングされうる値: タイムアウト・リトライ回数・バッチサイズ・ページサイズ・閾値・
  キャッシュ TTL・同時実行数・モデル名/デプロイ名
- 環境ごとに切り替えたい機能フラグ的な真偽値リテラル
- 同じリテラル値が複数箇所に散らばっているもの (1箇所の変更漏れ事故の元)

**見つけるもの (→ 名前付き定数化で十分。設定化はしない):**
- 数学・プロトコル定数 (86400, 1024, HTTP ステータスコード)、単位変換係数
- ドメイン上固定の値 (例: 週は7日)

**flag しないもの:**
- 0, 1, -1, 2 などの自明な値 (配列 index、増分、半分にする除数)
- テストコード内の期待値・フィクスチャ
- 一度しか使わない UI 文言 (i18n 対応はこのスキルのスコープ外。ただしユーザー向け文言に
  環境依存値が埋まっている場合は flag する)
- CSS の数値 (デザイントークン化はデザインシステムの仕事であり別タスク)

**重大度の目安:** 秘密情報 (API キー・パスワード) のハードコーディングは常に severity=high として
独立に報告する (→ 次元 9 hygiene と重複してよい。見逃しの方が問題)。

## 3. dry — 重複コード・共通化

**見つけるもの:**
- 3箇所以上に現れる実質同一のロジック (rule of three)
- 2箇所でも、完全同一 + 片方だけ修正される事故が起きそうなもの (バリデーション・変換・計算ロジック)
- コピペ後に片方だけバグ修正された形跡 (微妙に違う重複 — これは bug の温床なので severity 高め)
- 同じ外部 API / DB アクセスの呼び出し定型が util 化されず散らばっている
- 既存の共通 util があるのに使われていない再実装

**flag しないもの:**
- 偶然似ているだけで、ドメイン上は別の概念 (将来別方向に変更されるもの)。
  「この2つは同時に変更されるべきか?」で判断する。No なら統合しない
- テストコードの setup の重複 (適度な重複はテストの可読性に寄与する。
  ただし fixture/factory 化で明確に改善するものは flag してよい)
- 2行程度の些細な重複 (統合の間接化コストの方が高い)

**修正提案には共通化先を含める:** プロジェクトの慣例の置き場 (utils/, lib/, shared/, hooks/ 等) を
特定し、`proposed_fix` に「どこに・どんなシグネチャで抽出するか」まで書く。

## 4. structure — 構造・ベストプラクティス

**見つけるもの:**
- レイヤリング違反 (例: API 層から repository 直呼びでサービス層をバイパス、
  UI コンポーネントからの直接 fetch でデータ層をバイパス)
- 循環依存
- 神クラス / 巨大ファイル / 巨大関数 (目安: 関数 80 行超・ファイル 800 行超で要検討。
  ただし行数自体ではなく「責務が複数混ざっているか」で判断する)
- 責務の混在 (1つの関数が取得・変換・保存・通知を全部やる等)
- フレームワークのアンチパターン (そのプロジェクトのフレームワークのベストプラクティスに照らす。
  例: React の useEffect 乱用、FastAPI の依存性注入を使わないグローバル状態)
- 抽象化の漏れ (provider/adapter パターンがあるのに特定ベンダー実装が漏出している等)

**flag しないもの:**
- 動いていて閉じている古いコードの「今風でない」だけの書き方 (改善益が薄い)
- 意図的なパフォーマンス最適化による複雑さ (コメントや履歴に意図があるもの)

**リスク評価:** 構造変更は原則 risk=medium 以上、多くは behavior_change=possible。
大改修になるものは「見送り推奨 + 別タスク化」としてレポートに載せ、このブランチでは触らない。

## 5. dead-code — デッドコード・未使用コード

**見つけるもの:**
- どこからも参照されない関数・クラス・コンポーネント・hook・型定義
- 未使用 import / 未使用変数 / 未使用引数
- 呼ばれないエンドポイント・ルート (フロントエンドからの参照を Grep で確認)
- 常に false/true になっていて到達しない分岐、無効化されたままの機能フラグの残骸
- コメントアウトされたまま放置されたコードブロック (git 履歴にあるので消してよい)
- 使われていない設定キー・環境変数定義
- exports されているがプロジェクト内外から使われていないもの

**必ず Layer C 検証を通すこと。** 静的参照ゼロでも以下の経路で「生きている」ことがある:
- 文字列ベースの動的参照 (`getattr`, `globals()`, 動的 import, ルート名文字列, イベント名)
- DI コンテナ / プラグイン機構 / デコレータによる自動登録 (FastAPI ルーター、Celery タスク、
  pytest fixture/conftest, Alembic マイグレーション)
- 設定ファイル・テンプレート・YAML/JSON からの参照 (CI 設定、docker-compose の command、
  package.json scripts)
- 公開ライブラリ / 外部リポジトリの消費者 (パッケージとして publish されているものの public API)
- フレームワークの規約ベース解決 (Next.js の app/ 配下のファイル名規約、マイグレーションファイル)

**flag しないもの:**
- 将来用と明示されたインターフェースの未実装メソッド (ただし TODO 棚卸しには載せる)
- テストからのみ参照されるテストヘルパー (それは生きている)

**ツールでの裏取り:** 利用可能なら ts-prune / knip (TS), vulture (Python), ruff の F401/F841,
eslint no-unused-vars, depcheck (npm) 等を実行して突合する。ツール出力にも偽陽性はあるので
「ツールと精読の両方が未使用と判断 + Layer C で反証なし」のものだけ削除対象とする。

## 6. error-handling — エラーハンドリング・ロギングの一貫性

**見つけるもの:**
- 握りつぶし (`except: pass`, 空 catch, エラーを無視した Promise)
- 同種の失敗に対する処理方法の分裂 (片方は例外送出、片方は None 返却)
- ログレベルの誤用 (エラーを info で出す等)、ログの出し忘れ・二重出力
- ユーザー向けエラーメッセージへの内部情報 (スタックトレース・SQL) の漏出
- リソースリーク (close されないファイル/接続、context manager / finally の不使用)

**flag しないもの:**
- 意図的な best-effort 処理でコメントにその旨があるもの

## 7. types — 型安全

**見つけるもの:**
- TypeScript: `any` の濫用、`as` の強制キャスト、`!` non-null assertion の乱用、
  strict で回避できるのに ts-ignore している箇所
- Python: 公開関数の型ヒント欠落 (プロジェクトが型ヒント運用をしている場合)、
  `# type: ignore` の濫用、Any の濫用
- 型と実態の乖離 (optional なのに non-null 前提のコード)

**flag しないもの:** プロジェクトが型を運用していない領域への新規導入提案 (それは別タスク)。

## 8. deps — 依存パッケージの衛生

**見つけるもの:**
- 宣言されているが import されていない依存 (depcheck / pip での突合)
- 同じ役割のライブラリの重複 (lodash と ramda、axios と fetch ラッパー併存等)
- dependencies / devDependencies の誤配置

**flag しないもの:** ビルド・ランタイムが暗黙に要求する依存 (peer、プラグイン)。削除前に必ず検証。
バージョンアップ提案はこのスキルのスコープ外 (挙動が変わるため。報告のみ)。

## 9. security — セキュリティ

**見つけるもの:**
- インジェクション: SQL (文字列結合クエリ)、コマンド (shell=True + 外部入力)、パストラバーサル
  (外部入力をパス結合)、SSRF (外部入力 URL への fetch)、テンプレートインジェクション
  (外部入力をテンプレート文字列として評価)
- 安全でない評価・デシリアライズ: 外部入力が届きうる `eval` / `Function` コンストラクタ /
  `pickle.loads` / `yaml.load` (Loader 未指定) / 安全でない JSON 以外のデシリアライザ
- XML/XXE: 外部エンティティ解決が有効な XML パーサへの外部入力
- 認証・認可の穴: 認可チェックのないエンドポイント、IDOR (他人のリソース ID を検証なしで受理)、
  トークン検証の不備 (署名未検証・有効期限未チェック・aud/iss 未検証・alg=none 許容)、
  オープンリダイレクト (外部入力をリダイレクト先に使用)
- マスアサインメント: リクエストボディをフィールド制限なしでモデル/DB 更新に直結
- 安全でない暗号: MD5/SHA1 のパスワード用途、固定 IV・固定ソルト、`random` の秘密用途
  (secrets モジュール / crypto.randomBytes を使うべき箇所)、秘密鍵・署名鍵のハードコード
- 入力検証の欠如: サイズ無制限の入力受理、Content-Type 未検証のアップロード、
  外部入力に対する破滅的バックトラッキングを起こしうる正規表現 (ReDoS)
- 機微情報の露出: ログへのトークン/パスワード出力、エラーレスポンスへの内部情報漏出、
  CORS の `*` 許可 (credentials 併用)、デバッグモードの本番有効化
- 依存の既知脆弱性: ツール事前スキャン (osv-scanner / npm audit / pip-audit 等 — tooling.md 参照) の
  結果を「実際にその脆弱な経路を使っているか」で選別して報告する。
  修正はバージョンアップ = 挙動変更なので**報告のみ** (このブランチでは直さない)

**flag しないもの:**
- 内部専用スクリプト・テストコードでの簡易な扱い (ただし本番コードに混入しうる場所なら flag)
- フレームワークが既に防いでいるもの (ORM のパラメータ化クエリを「SQL があるから」で flag しない)

**修正の扱い:** セキュリティ修正の多くは挙動変更 (今まで通っていた入力を拒否する等) になるため、
原則 behavior_change=possible/yes として「要個別判断」枠で報告する。自動修正してよいのは
ログからの機微情報除去など出力側の無害な修正のみ。severity は悪用可能性で判断し、
外部入力が届く経路が実在するものは high。

**専門エージェントの併用:** security-reviewer 系の subagent type がインストールされていれば、
この次元はそちらに任せる方が精度が高い (orchestration.md 参照)。

## 10. hygiene — その他の衛生

**見つけるもの:**
- 秘密情報のハードコーディング (API キー・トークン・パスワード) — 常に severity=high。
  git 履歴に残っている場合はローテーション推奨も報告する。
  **値は絶対に findings・レポートに転記しない** — file:line とキー名、先頭数文字 + マスクのみ
  (SKILL.md 絶対原則 5)。発見時点でユーザーへ即時報告する (絶対原則 6)。
  ツール事前スキャン (gitleaks 等 — tooling.md) の結果と精読を突合する。
  `.env` など untracked ファイル内のものは正しい配置なので flag しない
- TODO / FIXME / HACK の棚卸し (消化済みなのに残っているもの、チケット番号のない放置 TODO)
- ドキュメントとコードの乖離 (README のコマンドが動かない、コメントが実装と矛盾)
- デバッグ残骸 (print / console.log の置き忘れ)

**flag しないもの:** 意図的なロギングとしての console 出力 (CLI ツールの出力等)。
