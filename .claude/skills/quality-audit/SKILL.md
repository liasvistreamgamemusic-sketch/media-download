# Quality Audit — 包括コード品質監査 & 安全リファクタリング

プロジェクトを複数エージェントで分割して全ファイルを精査し、findings をレポート化してユーザー承認を得たうえで、
リファクタブランチ上でカテゴリ単位に修正・テスト・コミットしていくスキル。

## 絶対原則

1. **機能を壊さない。** 自動で実行してよいのは挙動を変えない (behavior-preserving) 修正のみ。
   挙動が変わる可能性のある修正は必ずレポートの「要個別判断」枠に分離し、承認時に明示する。
   なぜなら、このスキルの価値は「安心して任せられること」にあり、一度でも機能を壊せばその信頼は失われるため。
2. **1ファイルも漏らさない。** カバレッジマニフェスト(後述)で全対象ファイルの監査済みを機械的に検証する。
   「だいたい見た」は禁止。未カバーが残っていたら追加監査する。
3. **テストゲートを通らない変更はコミットしない。** 各カテゴリの修正後に lint → build → test を実行し、
   グリーンであることを確認してからコミットする。
4. **新しい規約を発明しない。** 統一の基準は常に「そのプロジェクトの既存の多数派スタイル」と
   「プロジェクトが明文化した規約 (CLAUDE.md / .claude/rules/ / lint 設定)」。自分の好みを持ち込まない。
5. **秘密情報を拡散しない。** 監査中に発見した API キー・トークン・パスワードの「値」は、
   レポート・findings JSON・エージェントへのプロンプト・コミットメッセージのどこにも転記しない。
   記録するのは場所 (file:line)・キー名・マスク済みの先頭数文字 (`sk-live-ab****`) のみ。
   発見自体がインシデントであり、監査成果物への転記は流出面を広げる二次被害になるため。
6. **Critical は即時報告。** 悪用可能な脆弱性・有効な可能性のある secrets を見つけたら、
   レポート完成 (Phase 4) を待たずその時点でユーザーに知らせる (マスクした形で)。
   ローテーション等の初動は監査の完了より優先されるため。

## 引数の解釈

ユーザーの指示から以下を読み取る。曖昧なら確認する:

- **scope**: 監査対象。デフォルトはリポジトリ全体。「backend だけ」「frontend/lib 配下」等の指定があればそれに限定。
- **mode**: `full`(監査→承認→修正→コミット、デフォルト) / `dry-run`(監査とレポートのみ。ブランチ作成も修正もしない)。
  「ドライラン」「レポートだけ」「まず洗い出して」等の表現は dry-run。
- **incremental**: 過去の監査ワークスペース (`.quality-audit/`) が存在し、ユーザーが「差分だけ」「前回以降の変更だけ」
  と指示した場合は、前回監査コミット以降の `git diff --name-only` で変更ファイルのみを Layer A の対象にする。
  ただし横断次元 (dry / conventions) は変更ファイルが触れる相手側も読む必要があるため、変更ファイル + その参照先に広げる。
  レポートには「差分監査 (基準: <commit>)」と明記し、全体保証ではないことを示す。
- **追加の観点**: ユーザーが特定の関心 (例: セキュリティ重視、パフォーマンス) を示したら、該当次元の優先度を上げる。

## 全体フロー

```
Phase 0  前提確認 (規約収集・ベースライン確立)
Phase 1  リファクタブランチ作成            ← dry-run では skip
Phase 2  インベントリ作成 (カバレッジマニフェスト)
Phase 3  マルチエージェント監査 (ツール事前スキャン → fan-out+検証 → カバレッジ検査)
Phase 4  レポート提示 & ユーザー承認ゲート  ← dry-run はここで終了
Phase 5  修正フェーズ (カテゴリ単位: 修正 → テストゲート → コミット)
Phase 6  最終検証 & 完了報告
```

長時間の作業になるため、Phase 3 以降の中間成果物 (findings JSON、レポート、進捗) は
ワークスペース `<repo>/.quality-audit/<YYYYMMDD>/` に保存し、中断しても再開できるようにする。
このディレクトリは `.git/info/exclude` に追加してリポジトリを汚さない (tracked な .gitignore は変更しない)。

## Phase 0: 前提確認

1. **git 状態の確認**: working tree が clean か確認。dirty なら停止してユーザーに確認する
   (勝手に stash しない — ユーザーの作業中の変更かもしれない)。
2. **規約の収集**: 以下を読み、プロジェクトの「正」を把握する。
   - CLAUDE.md / .claude/rules/ 配下 (コミット形式・ブランチ命名・言語規約)
   - lint / formatter 設定 (ruff.toml, pyproject.toml, .eslintrc*, biome.json, prettier 設定, editorconfig 等)
   - 既存コードのイディオム (代表的なファイルを数件読む)
3. **検証コマンドの特定**: テスト・lint・build の実行コマンドを特定する。
   探索順: CLAUDE.md → package.json scripts → Makefile/justfile → CI 設定 (.gitlab-ci.yml, .github/workflows)。
   見つからなければユーザーに確認する。
4. **ベースライン実行**: 特定したテスト・lint・build を1回実行し、結果を記録する。
   既に失敗しているものがあれば「既存の失敗」として記録する — 後で自分の変更による失敗と区別するため。
   ベースラインが大きく壊れている場合 (テストの過半が落ちる等) は、監査より先にその修復を提案する。
   dry-run モードではコマンドの特定のみ行い、実行はスキップしてよい (修正しないので比較対象が不要)。

## Phase 1: リファクタブランチ作成 (dry-run では skip)

- デフォルトブランチ (main 等) の最新から作成する。
- ブランチ命名はプロジェクト規約に従う。チケット番号運用 (例: `refactor/{番号}-{kebab-case}`) があるプロジェクトでは
  ユーザーにチケット番号を確認する。規約がなければ `refactor/quality-audit-YYYYMMDD`。

## Phase 2: インベントリ作成

1. `git ls-files` で全 tracked ファイルを取得し、監査対象外を除外する:
   生成物 (lock ファイル、ビルド成果物、マイグレーションの自動生成部、`*.min.*`)、vendored コード、バイナリ/画像。
   除外したものはマニフェストに「除外 + 理由」として残す (漏れではなく意図的除外であることを示すため)。
2. 対象ファイルを**チャンク**に分割する。ディレクトリ・レイヤー単位でまとめ、1チャンク = エージェント1体が
   精読できる量 (目安: 20〜40ファイル or 3,000〜5,000行。巨大ファイルは単独チャンクでよい)。
3. マニフェスト (`manifest.json`: 全ファイル → チャンク ID の対応表) をワークスペースに保存する。

## Phase 3: マルチエージェント監査

詳細な編成・プロンプト・findings スキーマは [references/orchestration.md](references/orchestration.md) を読むこと。
監査の各次元のチェック基準と偽陽性ガイドは [references/audit-dimensions.md](references/audit-dimensions.md) を読み、
エージェントへのプロンプトに該当部分を含めること。

**3.0 ツール事前スキャン** (エージェント fan-out の前に必ず実施):
[references/tooling.md](references/tooling.md) を読み、そのエコシステムで使える静的ツール
(lint / 型チェック / デッドコード検出 / 未使用依存 / secrets スキャン / 依存脆弱性監査) を一括実行し、
結果をワークスペース `tools/` に保存する。機械が確実に検出できるもの (未使用 import、既知 CVE、secrets パターン)
をツールに任せることで、エージェントは「ツール所見の検証」と「機械では見えない文脈判断」に集中でき、
速度と精度が同時に上がる。ツール出力は該当チャンクごとに分配し、Layer A のプロンプトに「ツール所見」として添付する。
ツール出力にも偽陽性はあるため、finding の根拠の一つにはするが、それ単独を自動修正の根拠にしない。

構成は4層:

- **Layer A — チャンク監査** (チャンクごとに1エージェント、並列):
  担当チャンクの全ファイルを精読し、全次元 (規約逸脱・マジックナンバー/ハードコーディング・デッドコード候補・
  構造問題・エラーハンドリング・型安全・セキュリティ 等) の findings を構造化して返す。
  ツール所見は精読と突合し、「昇格 (finding 化)」か「偽陽性 (棄却理由つき)」に振り分ける。
  加えて「チャンク内で見つけたユーティリティ的処理のシグネチャ一覧」を返す (Layer B の重複検出の材料)。
- **Layer B — 横断監査** (次元ごとに1エージェント、並列):
  ファイル横断でしか見えない問題を担当する。①重複コード検出 (Layer A のシグネチャ集約 + Grep で裏取り)、
  ②規約ゆらぎの全体統一感 (命名・import 順・エラー処理スタイルの多数派特定)、③レイヤリング/依存方向の評価、
  ④設定・環境変数管理の一貫性。
- **Layer C — 偽陽性検証** (finding 単位、並列):
  検証対象は ①デッドコード候補、②重複統合候補、③severity=high のセキュリティ finding。
  デッドコードは動的参照 (文字列参照・設定キー・DI 登録・ルート登録・リフレクション・テンプレート参照) を
  リポジトリ全体 Grep で確認。セキュリティは「その入力経路は本当に外部から届くか」「フレームワークが既に
  防いでいないか」「そのコードパスは本番で到達可能か」を反証として確認する。
  確証が持てないものは「削除/high」から「報告のみ/needs-human」に降格する。
  重複は必ずマージする — Layer A と B が同じ問題を別々に報告することがあるため、
  file + line 近接 + 同一 dimension のものは集約時に1件へ統合し、verdict が割れていたら
  厳しい方 (keep > needs-human > confirmed) を採る。同一チャンク内の重複は検証に回す前に統合する。
- **Layer D — カバレッジ検査**:
  全 Layer A エージェントが返した `files_reviewed` をマニフェストと突合し、未カバー 0 を確認する。
  漏れがあれば追加エージェントを出して埋める。この結果 (対象 N 件 / 監査済み N 件 / 除外 M 件) はレポートに記載する。

実行手段: Workflow ツールが利用可能ならそれを使う (このスキルの指示による呼び出しは明示的オプトインに該当する)。
なければ Agent ツールの並列起動で同じ構成を組む。エージェント数はチャンク数に応じて自然に決まる —
無理に減らさない (漏れの原因になる) が、1チャンクを複数エージェントに重複させもしない。

**待ち時間を作らない:** Layer C の検証は「全チャンク完了」を待つ必要がない — あるチャンクの findings は
そのチャンクが終わった時点で検証を始められる (orchestration.md のパイプライン構成を使う)。
バリアが必要なのは Layer B (全チャンクのシグネチャ集約が入力) だけ。
監査エージェントの reasoning effort は下げない (読み落としが増え、監査では偽陰性が最も高くつく)。

環境に専門レビュアー (code-reviewer / security-reviewer / 言語別レビュアー等の subagent type) が
インストールされていれば、汎用プロンプトより精度が高いので該当次元・該当言語の監査に併用する
(詳細は orchestration.md の「専門エージェント・他スキルの併用」)。

## Phase 4: レポート提示 & 承認ゲート

findings を集約し、レポートをワークスペースに保存したうえで、要約をユーザーに提示する。構成:

```markdown
# コード品質監査レポート — <project> (<date>)
## サマリ          … 次元 × 重大度の件数表、カバレッジ実績 (対象/監査済/除外)
## カテゴリ別 findings … 各 finding: ID / file:line / 内容 / 修正方針 / リスク / 挙動変更有無
## 要個別判断       … 挙動が変わる可能性のある修正 (理由つき)
## 修正計画         … カテゴリの実施順序とコミット予定メッセージ
## 見送り推奨       … 直さないことを推奨するもの + 理由 (偽陽性疑い・費用対効果・別タスク化すべき大改修)
```

scope が限定されている場合、横断監査が scope 外で見つけた問題 (重複の相手側・呼び出し元の不備等) は
削除せず「スコープ外 (参考)」として独立セクションに載せる — 修正対象には含めないが、
全体監査や別タスクの種として価値があるため。

そのうえで **AskUserQuestion で承認を取る**: 「全て修正 (要個別判断も含む)」「安全な修正のみ (推奨)」
「カテゴリを選んで修正」の選択肢を出す。承認なしに Phase 5 へ進んではならない。
dry-run モードはここで終了し、レポートのパスを伝える。

## Phase 5: 修正フェーズ

修正方法のカテゴリ別ルールは [references/fix-policies.md](references/fix-policies.md) を読むこと。

**実施順序** (前のカテゴリが後のカテゴリの作業量を減らす順):

1. 秘密情報の除去 — インシデント性があるため最優先。ローテーション推奨とセットで
2. デッドコード削除 — 消すものを先に消す。消えるコードを磨くのは無駄なため
3. 規約統一 — formatter / lint --fix の自動修正を最大限使い、残りを手動で
4. マジックナンバー / ハードコーディングの設定化 — 既定値は現行値、挙動不変
5. 重複コードの共通化
6. エラーハンドリング・型安全・ロギングの一貫性修正
7. 構造改善 — 承認済みのもののみ。大改修は別タスク化を提案

**各カテゴリの手順** (このループを崩さない):

```
修正実施 → lint → build → test (ベースラインの既存失敗は除外して判定)
  → グリーン: そのカテゴリのみ git add してコミット (1カテゴリ = 1コミット)
  → レッド:  原因を特定して修正。3回試して解決しなければ該当変更を revert し、
            「見送り (理由: ...)」としてレポートに追記して次のカテゴリへ
```

- コミットメッセージはプロジェクト規約に従う (Conventional Commits、プロジェクトの言語、チケット参照等)。
  例: `refactor: マジックナンバーを設定値へ抽出 refs #1234`
- risk=medium 以上のカテゴリ (共通化・構造改善) は、コミット前に code-reviewer 系エージェントや
  `/code-review` 相当のスキルが使えるなら diff レビューを挟む — テストで捕まらないロジック退行の検出に有効。
- 1カテゴリ = 1コミットにする理由: レビューしやすく、問題があったとき `git revert` で
  カテゴリ単位に安全に巻き戻せるため。
- 修正のエージェント分担は可能だが、**同一ファイルを複数エージェントが同時に触らない**ようにファイル単位で分割する。
- ワークスペースのレポート類は絶対に `git add` しない。

## Phase 6: 最終検証 & 完了報告

1. フルで lint・build・テストを再実行。プロジェクトに起動手段 (docker compose 等) があれば起動確認まで行う。
2. 完了報告: コミット一覧 / 修正した findings 数 / 見送った findings + 理由 / カバレッジ実績。
3. MR / PR の作成はプロジェクト規約 (Draft 運用等) に従い、ユーザーに確認してから行う。
4. ワークスペース (`.quality-audit/`) は残すか消すかをユーザーに確認する (再監査時の差分比較に使えるため)。

## してはいけないこと

- 承認前に修正を始める / 承認範囲外の修正を「ついでに」入れる
- テストが落ちたままコミットする、または失敗を握りつぶして次へ進む
- 確証のないデッドコード削除 (Layer C 検証を通っていないもの)
- 設定化の際にデフォルト値を「より良い値」に変える (挙動が変わる)
- tracked ファイルである .gitignore や lint 設定をユーザーへの説明なしに変更する
- ベースラインで既に落ちていたテストを「自分が直すべきもの」として深追いする (報告に留め、別タスクを提案)
- 発見した secrets の値をレポート・プロンプト・コミットメッセージに転記する (常にマスク)
- 監査用ツールをプロジェクトの依存 (package.json / pyproject.toml 等) に追加する
  (一時実行 — npx / uvx 等 — に留める。詳細は tooling.md)
- テスト・lint 以外でプロジェクトのコードを本番相当の設定・実データに対して実行する
