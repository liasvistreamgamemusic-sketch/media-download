# user-manual — 画面キャプチャから作るエンドユーザー向け操作マニュアル (PPTX)

実際に動いているアプリの画面を唯一の情報源として、コードを一切読まない利用者が
操作を完遂できるマニュアルを PowerPoint で生成する。
姉妹スキル project-manual (開発者向け・Markdown) とは読者も成果物も別物 —
このスキルの読者は「ボタンがどこにあるか」を知りたい人であり、仕組みを知りたい人ではない。

## 絶対原則

1. **撮影漏れゼロを機械的に保証する。** コードのルート定義から画面一覧 (ルートレジストリ) を列挙し、
   「レジストリの画面数 = 撮影済み画像数 = スライドに掲載した画面数」の突合で網羅を証明する。
   「主要そうな画面を撮った」は禁止 — 対象外にする画面は理由付きでレジストリに残す。
2. **スクリーンショットは全数目視検証する。** 撮影した画像を 1 枚ずつ Read で開いて確認し、
   白画面・ローディング中・ログイン画面へのリダイレクト・エラー表示・`undefined`/`NaN` の写り込みを弾く。
   NG は原因を直して撮り直すまで次工程に進まない (チェック観点: [references/capture.md](references/capture.md))。
3. **操作説明は事実だけを書く (FABLE Anchor)。** 手順・ボタン名・遷移先は、スクショとコード
   (ハンドラ・ルーティング) で確認したものだけ書く。UI に存在しない機能・未確認の挙動を
   書くのは絶対禁止 (Iron Rule 3)。確認できなかった動作は本文に載せず完了報告の未確認リストへ。
4. **デザイントークン駆動で作る。** 色・フォント・余白・レイアウトは 1 箇所のトークン定義に集約し、
   全スライドがそれだけを参照する。アクセント色はアプリ自身のブランド色を CSS から抽出して使う
   (マニュアルが製品の一部に見える)。実装前に [references/pptx-design.md](references/pptx-design.md) を必ず読む。
5. **編集可能なまま納品する。** コールアウト (番号丸・枠・注記) は画像に焼き込まず
   pptx のシェイプとして載せる。ユーザーが PowerPoint 上で文言・位置を直せることが納品条件。
6. **実データ・secrets を写さない。** 撮影は seed / デモデータで行う。実在の個人情報・トークンが
   写る場合は Playwright の mask で隠すか撮り直す。認証セッション (storageState) は
   ワークスペース内に置き `.git/info/exclude` で除外する。tracked ファイルに認証情報を書かない。

## 引数の解釈

- **対象範囲**: 「管理画面だけ」等の指定があればルートレジストリをその範囲に限定し、表紙に対象範囲を明記する。
- **出力先**: 指定があれば従う。なければ Phase 1 で確認する (デフォルト案: `docs/manual/操作マニュアル_<アプリ名>.pptx`)。
- **更新モード**: 過去のワークスペース (`.ui-manual/`) が存在し「更新して」「画面差し替えて」と指示されたら、
  前回以降の `git diff --name-only` からルート・画面の変更を特定し、変更画面のみ再撮影して該当スライドを差し替える。
  ルートレジストリの突合とページ番号・目次は必ず再生成する。

## 全体フロー

```
Phase 0  コンテキスト把握 (フレームワーク・ルート列挙・起動方法・認証)
Phase 1  ボタン質問 1 回 (範囲・読者・デザイン・出力)
Phase 2  撮影 (Playwright) + 全数目視検証
Phase 3  操作フロー抽出 (画面ごとの「できること」と手順)
Phase 4  PPTX 生成 (python-pptx・トークン駆動)
Phase 5  視覚セルフレビュー (pptx → 画像化して通読) + 網羅検査
Phase 6  納品 & 完了報告
```

中間成果物 (ルートレジストリ・スクショ・facts JSON・生成スクリプト) はワークスペース
`<repo>/.ui-manual/<YYYYMMDD>/` に保存し、中断しても再開できるようにする。
このディレクトリは `.git/info/exclude` に追加する (tracked な .gitignore は変更しない)。

## Phase 0: コンテキスト把握 (質問の前に必ず)

1. **フレームワーク検出とルート列挙**: [references/capture.md](references/capture.md) の検出表に従い、
   ルート定義からページ一覧を機械的に列挙して `routes.json` (ルートレジストリ) を作る。
   動的ルート (`[id]` 等) は代表データの URL が要る — seed / fixture から探し、なければ Phase 1 で聞く。
2. **起動方法の特定**: package.json scripts / README / compose から dev server の起動コマンドと URL を特定する。
   既に起動していれば (`curl` で確認) それを使う。自分で起動した場合は撮影後に必ず止める。
3. **認証の把握**: ルートガード / middleware / ログインページの有無から認証の要否を判定する。
   デモアカウントを `.env.example` / seed / README から探し、見つからなければ Phase 1 で聞く。
4. **ブランド色の抽出**: アプリの CSS variables / Tailwind config / テーマ定義から
   プライマリ色を 1 つ抽出しておく (Phase 1 の推奨ボタンに使う)。
5. **公式 pptx スキルの有無確認**: 利用可能なスキル一覧に `pptx` (Anthropic 公式・
   プラグイン `document-skills`) があるか確認する。Phase 4 の生成トラック選択に使う。
   なければ Phase 1 の質問に導入可否を含める。

## Phase 1: AskUserQuestion で 1 回だけ質問 (最大 4 問)

推測できるものは聞かない — 推測値を推奨ボタンにして確認だけ取る。引数で指定済みの項目は省く。

**質問を出す直前に、Phase 0 で列挙したルート一覧 (画面数と各画面のパス・名前) を通常の
テキストとして必ず表示する**。preview はフォーカスした選択肢しか表示されない環境があるため、
撮影対象の確認を preview だけに頼ってはならない — 一覧が読める状態にしてから質問する。

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | **対象範囲** | ルート一覧は質問前に本文表示したうえで preview にも載せ、「全画面 (推奨・N 画面)」/「主要フローのみ (ログイン〜基本操作)」/「範囲を指定したい」。ここが撮影対象の確認ゲート |
| 2 | **読者とトーン** | 「一般エンドユーザー (推奨: 敬体・専門用語なし)」/「社内オペレーター (業務用語 OK)」/「管理者向け (設定・権限中心)」 |
| 3 | **デザイン** | 「アプリのブランド色に合わせる (推奨: 抽出した色名 + hex)」/「指定のコーポレートカラー」/「モノクロ基調 (堅め)」 |
| 4 | **出力** | 「PPTX のみ (推奨)」/「PPTX + Markdown 版も」。出力先パスの確認もここに含める |

認証情報・動的ルートの代表 URL が Phase 0 で見つからなかった場合は、質問を 1 枠使って聞く
(その場合は優先度の低い質問を落とす)。

公式 pptx スキルが未導入の場合は質問を 1 枠使い、「公式 pptx スキルを導入して生成
(推奨 — claude.ai の PowerPoint 生成と同系統でデザイン品質が高い。導入:
`/plugin marketplace add anthropics/skills` → `document-skills` をインストール)」/
「python-pptx フォールバックでこのまま進める」を確認する。

## Phase 2: 撮影 + 全数目視検証

[references/capture.md](references/capture.md) のレシピに従う。要点:

- Playwright (chromium)・viewport 1440×900・`deviceScaleFactor: 2` (Retina 画質)・`locale: 'ja-JP'`。
- アニメーション無効化 CSS を注入し、`networkidle` + `document.fonts.ready` を待ってから撮る (ブレ防止)。
- 認証はログインフローを 1 回実行して `storageState` を保存し、以降のページで再利用する。
- 各ルートで **ビューポートショット (スライド掲載用)** と **フルページショット (参考用)** の 2 枚を撮る。
- 同時に、その画面の主要 UI 要素 (ボタン・フォーム・ナビ) の **boundingBox を JSON で記録**する —
  Phase 4 のコールアウト配置にピクセル座標として使う (目分量で置かない)。
- 撮影後、全画像を Read で開いて目視検証する (原則 2)。検証結果はレジストリに `verified: true/false + 理由` で記録。

## Phase 3: 操作フロー抽出

画面ごとに「利用者が何をできるか」を確定し、`facts.json` に構造化する:

- **画面の目的** (1 文)・**主要素** (ボタン/フォーム/一覧とその役割)・**操作手順** (操作 → 結果の連鎖)・
  **遷移先** (どのボタンでどの画面へ)。
- 根拠はスクショ + コード (onClick ハンドラ・form action・ルーティング) の両方。コードだけから
  「たぶんこう動く」を書かない — 表示に現れない挙動は手順にしない。
- モーダル・タブ・入力エラー表示など**操作しないと現れない状態**が手順に必要なら、
  この段階で追加撮影する (Playwright で操作を再現してから撮る)。追加分もレジストリと目視検証に載せる。
- 手順の文章は [references/manual-structure.md](references/manual-structure.md) の文体規則に従う
  (1 手順 1 動作・「操作します → 表示されます」形式・専門用語ブラックリスト)。

## Phase 4: PPTX 生成 (2 トラック)

**Track A (推奨) — Anthropic 公式 pptx スキル**が導入済みならそれで生成する
(claude.ai の PowerPoint 生成と同系統。PptxGenJS ベースでデザイン品質が高い):

- 公式スキルの SKILL.md と作成ガイドを読み込み、その流儀で生成スクリプトを書く。
- ただし**本スキルの要件が常に優先**: 章構成・文体は
  [references/manual-structure.md](references/manual-structure.md) / スクショの縦横比厳守 /
  コールアウトは Phase 2 の boundingBox から座標計算した編集可能シェイプ /
  アクセント色はアプリのブランド色 / 日本語フォントを全テキストに明示
  (これらの要件定義: [references/pptx-design.md](references/pptx-design.md) §0)。
- 公式スキルのデザイン提案 (パレット・レイアウト) は「操作マニュアル」の文脈で取捨する —
  営業資料的な演出 (巨大な数字・ヒーロー画像) より、画面の見やすさと手順の追いやすさが優先。

**Track B (フォールバック) — python-pptx**。公式スキルが使えない環境
(オフライン・プラグイン導入不可) 用。[references/pptx-design.md](references/pptx-design.md) を
読んでから書く。要点:

- 実行環境: `uv` があれば `uv run --with python-pptx build_pptx.py`。なければワークスペース内に
  venv を作って `pip install python-pptx` (グローバル環境を汚さない)。
- 冒頭でトークン定義 (色・フォント・余白・グリッド) を宣言し、全レイアウト関数はトークンだけを参照する。
- 日本語フォントは `a:ea` (East Asian typeface) を XML で明示設定する — `run.font.name` だけでは
  日本語が既定フォントに落ちる (pptx-design.md にヘルパーあり)。

どちらのトラックでも、生成スクリプトはワークスペースに保存する (差分更新で再利用する)。

## Phase 5: 視覚セルフレビュー + 網羅検査

1. **視覚検査**: LibreOffice (`soffice --headless --convert-to pdf`) があれば pptx を PDF 化し、
   全ページを Read で通読する — 文字あふれ・画像のアスペクト崩れ・コールアウトのずれ・
   フォント落ち (日本語が明朝/Times になっていないか) を確認して修正する。
   LibreOffice がなければ python-pptx で再オープンしてスライド数・テキストあふれの機械検査のみ行い、
   視覚未検証である旨を完了報告に明記する (検証したふりをしない)。
2. **網羅検査**: ルートレジストリの全画面がスライドに載っているか数を突合する。
   対象外にした画面が理由付きで報告に載っているか確認する。
3. **文体検査**: manual-structure.md の専門用語ブラックリスト・文体規則で全テキストを検査する。
4. **secrets / 実データ検査**: 全スクショに実在の個人情報・トークン・内部 URL が写っていないか再確認する。

## Phase 6: 納品 & 完了報告

1. 承認された出力先に `.pptx` (と指定があれば Markdown 版) を書き出す。コミットはユーザーの指示があれば
   git-commit スキルの流儀で行う。自分で起動した dev server を止める。
2. 完了報告 (日本語・結論から): 出力パス / 撮影・掲載実績 (対象 N 画面・撮影 N・掲載 N・対象外 M と理由) /
   目視検証の結果 (視覚検査を実施できたか) / 未確認事項リスト / 更新方法 (「画面変更後は差分更新できる」)。
3. ワークスペース (`.ui-manual/`) は差分更新の基準になるため、残すか消すかをユーザーに確認する。

## してはいけないこと

- スクショとコードで確認していない操作手順・ボタン名・遷移を書く
- 撮影に失敗した画面を黙って落とす (必ず理由付きで報告する)
- 実ユーザーデータ・secrets・内部ホスト名が写ったスクショをそのまま使う
- コールアウトを画像に焼き込む (シェイプで載せる — 原則 5)
- トークン定義なしで書き始める / Office 既定テーマのまま生成する
- `run.font.name` だけ設定して日本語フォントを既定に落とす (`a:ea` を必ず設定)
- スクショを縦横比を無視して引き伸ばす
- 自分で起動した dev server・ブラウザプロセスを放置する
- storageState / 認証情報を tracked ファイルに書く
- 視覚検査をしていないのに「レイアウト確認済み」と報告する (Iron Rule 2)
