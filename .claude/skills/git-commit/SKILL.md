# git-commit — ボタンで完結するコミット

## 手順

### 1. 状態把握と分析 (質問の前に必ず)

```bash
git status && git diff && git diff --cached   # 全変更を把握 (未追跡ファイルも見る)
git log --oneline -5                          # 直近のメッセージスタイルと最終コミット日時
```

- diff に秘密情報 (API キー・トークン・パスワード・`.env` の中身) があれば**質問せず停止して報告**
- 変更内容から **種別 (type) とコミットメッセージ案**を推測して作っておく
- ブランチ名にチケット番号が含まれていれば (例: `feat/1234-...`) メッセージ末尾に
  `refs #1234` を付けた案にする (Redmine 連携 — redmine-start が作るブランチはこの形式)
- 関心事が複数混ざっていれば分割案を作る (質問の選択肢に含める)
- メッセージ規約は `git-workflow.md` 準拠: `<type>: <description>`、本文は「なぜ」、Attribution なし

### 2. AskUserQuestion で 1 回だけ質問 (最大4問をまとめて)

**質問を出す直前に、生成したコミットメッセージ案の全文を通常のテキストとしてコードブロックで
必ず表示する** (例: 「以下のメッセージでコミットします:」+ メッセージ全文)。
preview はフォーカスした選択肢しか表示されない環境があるため、preview だけに
メッセージを載せてはならない — ユーザーが選択肢を見た時点でメッセージ本文が
読める状態にしてから質問する。

| # | 質問 | 選択肢 (先頭=推奨) | 備考 |
| --- | --- | --- | --- |
| 1 | メッセージ | 「この案でコミット (推奨)」(description に 1 行目を記載、preview に full メッセージ) / 「修正したい」 | 質問文にもメッセージ 1 行目を含める (例: 「`feat: …` でコミットしますか?」)。修正は「その他」に本文を入力してもらう |
| 2 | 種別 | 推測した type を先頭に「(推奨)」付きで + 代替 2〜3 個 | 案のままなら質問1と整合させる |
| 3 | コミット日時 | 「今の時刻 (推奨)」 / 「直前コミットの直後」 | 説明に「日時指定は『その他』に `18:30` や `2026-07-02 18:30` 形式で入力」と書く |
| 4 | push | 「コミットのみ (推奨)」 / 「push もする」 | 追跡ブランチがない場合は `-u origin HEAD` で push |

- 変更が複数の関心事なら質問1の選択肢に「2 つに分割してコミット」を加える (preview に分割案)
- 推測できるものは聞かない — 推測値を推奨ボタンにして確認だけ取る

### 3. 実行

```bash
git add <対象ファイルのみ>          # -A / . は使わず対象を明示
git commit -m "<確定メッセージ>"
```

日時指定があった場合 (author と committer の両方に同じ値):

```bash
GIT_AUTHOR_DATE="2026-07-02 18:30" GIT_COMMITTER_DATE="2026-07-02 18:30" git commit -m "..."
```

- 「`18:30`」のような時刻だけの入力 → 今日のその時刻。「直前コミットの直後」→ 最終コミット時刻 +1 分
- 相対表現 (昨日 18:30 等) は明示的な `YYYY-MM-DD HH:MM` に変換して渡す
- pre-commit hook が失敗したら原因を直して再コミット (`--no-verify` 禁止)。直せなければ失敗として報告

### 4. 報告 (日本語・結論から)

- 例: 「`fix: バッチ集計の丸め誤差を修正` をコミットしました (2026-07-02 18:30 付け・push 済み)」
- hook の失敗・コンフリクト等は隠さず先頭で、実際の出力とともに
