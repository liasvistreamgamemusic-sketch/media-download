# ui-craft — 実在ブランドの設計言語をアンカーに、シンプルかつリッチな UI を作る

成果物は 3 つ: **意図を持ってデザインされたと感じられる UI**、**ユーザーが安全に微調整できる構造**、
そして改善依頼では**見て分かる変化**。

## 中核原理 — ブランドアンカー

AI がゴミデザインを作る根本原因は「抽象的な原則から毎回ゼロで発明する」こと。
このスキルは発明しない — `references/brands/` に実在ブランド 73 種のデザインシステム実測データ
(実 hex・タイポスケール・余白・角丸・深度・コンポーネント仕様・Do/Don't・レスポンシブ挙動) があり、
**依頼のドメインに合うブランドを 1 つ選び、その設計言語に忠実に組み立てる**。
プロが何千時間もかけて磨いた一貫性を最初から借りる。

借用ルール (非交渉):

- 借りるのは**設計言語だけ** — 色・タイポ・余白・角丸・深度・コンポーネントの作法・Do/Don't。
  ロゴ・ワードマーク・商標・ブランド名の表記・固有イラストは**複製しない**
- コンテンツは依頼のドメインのもの。ブランドのコピーやプロダクト名を流用しない
- 商用フォントは各ブランドファイル末尾の "Font Substitutes" の無料代替
  (Inter / Geist / JetBrains Mono 等) を使う
- ブランドファイルの **Do's and Don'ts は絶対に守る** — ここを破ると「っぽさ」が消える
- 混ぜない。2 ブランドのいいとこ取りは一貫性を殺す。1 依頼 = 1 アンカー

## 絶対原則

1. **アンカー忠実。** 色・タイポ・角丸・余白・深度は選んだブランドファイルの実値をトークンに転記して
   使う。「だいたいこんな感じ」の自作値で薄めない。
2. **AI っぽさと物足りなさの両方を減点法で落とす。** `references/anti-ai-checklist.md` の
   ブラックリスト (過剰) と richness floor (不足) の両方に diff を突き合わせる。
3. **聞くべきことは実装前に 1 回で聞く。** AskUserQuestion 1 回 (最大 4 問)。
4. **トークン駆動。** 色・余白・角丸・フォント・モーションは 1 箇所のトークン定義に集約し、
   以後はトークンだけを参照する。
5. **モーションの既定はリッチ。** 動きのない UI は未完成 (ミニマル合意時のみ例外)。
   物理は transform/opacity のみ + `prefers-reduced-motion` 必須 (`references/motion.md`)。
6. **アクセシビリティは非交渉。** 本文系の色ペアは `scripts/validate-palette.mjs` で機械検証。
   FAIL したペアはブランドの ink ラダー内で 1 段濃い値に調整する (accent の装飾用途は対象外)。
   フォーカスリング・キーボード操作・セマンティック HTML を省かない。
7. **改善依頼は「変える」契約 (visible-delta contract)。** before/after 比較で視覚デルタが
   無ければ FAIL。保守が正しいのは拡張モードだけ。
8. **見ていない UI は納品しない。** スクリーンショットを撮り Read で確認するのが必須ゲート。
   撮影不能なときだけ「静的検証モード」を明示宣言してフォールバックする。

## モード判定 (最初に必ず)

| モード | 条件 | 基本姿勢 |
| --- | --- | --- |
| **新規** | UI がまだ無い / 独立した新ページ・新アプリ | ブランドアンカーを選んでゼロから設計 |
| **拡張** | 既存デザインシステムのあるプロジェクトへの新画面・新部品の**追加** | 保守が正しい — 既存トークン・語彙の中で作る。**ブランドアンカーは持ち込まない** (既存システムがアンカー) |
| **改善** | 既存画面への「改善して / リッチに / 見やすく / かっこよく / 今っぽく / 配色直して」 | **変えることが成果物** — 刷新先の設計言語としてブランドアンカーを提案。`references/redesign-mode.md` を読んで Phase 0.5 へ |

迷ったら改善に倒す (改善 Q1 の「トークン微調整」が保守側の意図を回収する)。
拡張モード中に既存 UI の問題を見つけたら、勝手に直さず報告で指摘する。

## 全体フロー

```text
共通     Phase 0    モード判定 + コンテキスト把握 + アンカー候補選定 (INDEX.md)
改善のみ Phase 0.5  現状診断 (トークン採取・before スクショ・ギャップリスト)
共通     Phase 1    ボタン質問 1 回 (最大 4 問)
共通     Phase 2    選定ブランドファイル読了 → トークン定義 → validate-palette.mjs
共通     Phase 3    実装 (アンカーのコンポーネント仕様 + motion.md のレシピ)
共通     Phase 4    検証ゲート (全 PASS まで納品しない)
共通     Phase 5    報告 (改善時は before/after 要約) + 微調整ガイド
```

### Phase 0 — コンテキスト把握 + アンカー候補選定

- 既存プロジェクト: トークン / テーマ / Tailwind config、スタイリング手法の多数派、既存ページの
  トーンを把握 (拡張=従うため、改善=診断して変えるため)。
- 新規: 目的・対象ユーザー・デバイスを読み取り、**情報設計ドラフト** (セクション構成・各セクションの
  要素) を作る。
- **`references/brands/INDEX.md` を読み、ドメイン・トーンに合うアンカー候補を 3〜4 件選ぶ**
  (新規・改善のみ)。ユーザーがブランド名や「〇〇風」に言及していたらそれを最有力候補にする。
  選定軸: ①用途の一致 (開発者ツール / SaaS / EC / メディア / 金融 / クリエイティブ) ②モード
  (dark/light) がプロダクト特性に合うか ③個性の距離 (無難すぎる候補ばかり並べない — 1 件は
  性格の強い対抗案を入れる)。

### Phase 0.5 — 現状診断 (改善モードのみ)

手順は `references/redesign-mode.md`。骨子: ①インベントリ採取 (スタック・トークン定義場所・現行実値)
②before スクリーンショット (1440×900、撮影不能なら静的検証モード宣言)
③ギャップ診断 (anti-ai-checklist 両方向と突き合わせ、`| # | 症状 | 箇所 | 改善案 | 期待効果 |` 表)。

### Phase 1 — AskUserQuestion で 1 回だけ質問 (最大 4 問)

共通ルール: 推測できるものは聞かず、推測値を推奨 (先頭ボタン) にして確認だけ取る。
質問の直前に、情報設計ドラフト (新規) / ギャップリスト要約 (改善) を**通常テキストで表示してから**聞く —
preview はフォーカスした選択肢しか表示されない環境があるため、確認対象を preview だけに載せない。
「お任せ」なら質問せず決定と理由を報告に含める。「その他」自由入力 (参考 URL・ブランド名) は最優先。

**新規モード**:

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | デザイン言語 (アンカー) | Phase 0 で選んだブランド候補 3〜4 件。**ラベル=「<brand> 系」、説明に canvas/accent の実 hex + 雰囲気 + そのドメインに合う理由 1 句** (INDEX.md の行から作る)。最適合を推奨先頭に。ダーク文脈・開発者ツールではダーク系 (linear.app / vercel / warp 等) を必ず 1 件含める |
| 2 | 表示要素・構成 | 直前に表示した情報設計ドラフトへの確認ゲート: 「この構成で OK (推奨)」/「調整したい」 |
| 3 | 演出のリッチ度 | **「リッチ (推奨)」**: 主役演出 + スクロール出現 + 素材感 + 全対話要素のマイクロ / 「スタンダード」: ロード演出 + ホバーのみ / 「ミニマル」: フェードのみ (管理画面向け)。説明に各レベルで何が動くかを書く |
| 4 | (必要時のみ) テーマ | アンカーのネイティブモードが 1 つで、依頼にもう片方の示唆があるときだけ「ネイティブのみ (推奨) / 両テーマ導出」を聞く。示唆が無ければ聞かずネイティブのみ |

**改善モード**:

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | 改善強度 | **「視覚リフレッシュ (推奨)」**: 構造は維持し色・タイポ・影・モーションを全面刷新 / 「全面リデザイン」: 情報設計から再設計 / 「トークン微調整」: 雰囲気を保ち数カ所だけ。説明に診断ギャップ上位 2〜3 件を織り込む |
| 2 | 刷新先デザイン言語 | 「<brand> 系に刷新 (推奨)」+ 実 hex / 対照的な第 2 ブランド / 「現行色を磨く」(現行 accent を保ちコントラスト・配分だけ再構成)。ブランド色 (企業 CI) は無断で変えない |
| 3 | 演出のリッチ度 | 新規 Q3 と同一 3 択 (リッチ推奨)。説明に診断で見つかった不足を書く |
| 4 | 適用範囲 | 「対象画面 + 共有基盤 (推奨)」/「対象画面のみ」/「サイト全体」。単一画面なら省略 |

**拡張モード**: デザイン言語・配色は聞かない (既存に従う)。構成 + リッチ度のみ。

### Phase 2 — アンカー読了 → デザイントークン定義

1. **選定された `references/brands/<brand>.md` を全文読む** (これが本実装の一次仕様書)。
2. frontmatter の colors / typography / rounded / spacing をプロジェクトのトークン定義に転記する:

```css
:root {
  /* color: ブランドの colors: をそのまま。命名はプロジェクト慣例に合わせてよいが値は実値 */
  --color-canvas: …; --color-surface-1: …; --color-ink: …; --color-ink-muted: …;
  --color-hairline: …; --color-accent: …; --color-on-accent: …;
  /* typography: ブランドのスケールから使う段だけ。フォントは代替指定に従う */
  --font-display: …; --font-body: …;
  /* spacing / radius: ブランドの spacing: / rounded: の実値 */
  /* motion: --duration-fast: 150ms; --duration-base: 250ms; --duration-slow: 450ms (詳細は motion.md) */
}
```

- Tailwind なら同内容を `@theme` / `theme.extend` に置く
- テーマは**アンカーのネイティブモードのみが既定** (Q4 で両テーマ合意時のみ導出 —
  導出手順は `references/color-system.md` §2)
- **定義直後に検証**: `node <スキルの場所>/scripts/validate-palette.mjs <tokens ファイル>`
  (`~/.claude/skills/ui-craft/scripts/` またはプロジェクトの `.claude/skills/ui-craft/scripts/`)。
  本文系ペアの FAIL はブランドの ink ラダー内で 1 段調整して再実行
- 改善モード (視覚リフレッシュ以下) では**既存トークン名を保って値を刷新**する
  (redesign-mode.md §5 — 呼び出し側変更ゼロで全面が変わる)
- アンカー無しで作る場合 (拡張モード / ユーザーがオリジナル配色を明示要求) のみ
  `references/color-system.md` のプリセットを使う

### Phase 3 — 実装

一次仕様書は**読了済みのブランドファイル** — レイアウト・余白・深度・コンポーネントは
その Components / Layout / Elevation / Do's and Don'ts 節に従う。追加で読むのは:

- `references/motion.md` — リッチ/スタンダード合意時は必読 (§6 のリッチ度マッピング表どおりに組む)
- `references/liquid-glass.md` — ガラス表現・Liquid Glass 言及時のみ
- `references/design-principles.md` + `references/rich-techniques.md` — **アンカー無しで作る時のみ**
  (アンカーがある時はブランドファイルがこの 2 つを代替する)

実装規律:

- レイアウトは情報の重要度から決める。**全要素等価のグリッドは思考放棄** — 1 画面の主役は 1 つ
- コンポーネントはまずブランドファイルの `components:` 仕様 (背景・文字色・角丸・padding) を
  そのまま組み、そこに合意リッチ度のモーションを重ねる
- ダミーではなく**そのドメインらしい現実的なコンテンツ** (「Lorem ipsum」「Feature 1/2/3」禁止)
- 画像プレースホルダは灰色ボックスでなく自作図版 (CSS グラデ・SVG パターン・擬似 UI)
- レスポンシブはブランドファイルの Responsive Behavior 節に従い、モバイル 375px を実確認

### Phase 4 — 検証ゲート (全 PASS まで納品しない)

1. **コントラスト機械検証**: validate-palette.mjs が exit 0
2. **スクリーンショット**: desktop 1440×900 + mobile 375px を撮影し **Read で自分の目で確認**。
   フォールバックチェーン: プロジェクトの playwright/puppeteer →
   `npx -y playwright screenshot --viewport-size=1440,900 <URL> <out.png>` → dev server は既存を
   `curl -sf` で優先、無ければ run_in_background で起動し終了時に停止 → 不可なら静的検証モード宣言
3. **アンカー忠実チェック**: ブランドファイルの **Do's and Don'ts に diff を突き合わせ**、
   Don't 違反ゼロを確認。あわせて `references/anti-ai-checklist.md` (richness floor 含む) も通す
4. **a11y / 挙動**: focus-visible・キーボード操作・`prefers-reduced-motion`・375px 崩れなし
5. **visible-delta 判定 (改善のみ)**: before/after を Read で比較 (redesign-mode.md §7)。
   視覚リフレッシュ = 6 観点中 3 以上 + 1 秒で違いが分かる、全面リデザイン = 5 観点以上。
   FAIL → Phase 2 に戻る (最大 2 周)

### Phase 5 — 報告 + 微調整ガイド

報告に必ず含める (日本語・結論から):

1. 何を作った/変えたか + **採用アンカー名**とデザイン意図 1〜2 文
2. 改善のみ: Before→After 要約表 + スクショ 2 枚の絶対パス + ギャップ消化状況
3. 検証結果: validate-palette PASS、Do's/Don'ts 通過、visible-delta 判定
4. **微調整ガイド**: 「色味 → `tokens.css` の `--color-accent`」の対応表を実パス・実トークン名で。
   末尾に「色を変えたら validate-palette.mjs で再検証」
5. 確認方法 (起動コマンド / ファイルパス)

### 納品後の修正リクエスト対応

- 「もう少し○○に」系 (色・速度・余白・角丸) → **トークンのみ変更**
- 要素の追加・削除 → 情報設計に影響するなら 1 問だけ確認、明確なら即実装
- 「別ブランド風に」→ Phase 2 からやり直し (トークン一式差し替えで構造は再利用)
- **「改善して」「リッチに」系の品質向上依頼 → 改善モードとして再入する**
