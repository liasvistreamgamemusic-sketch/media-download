# ui-craft — ぱっと見で分かる、シンプルかつリッチな UI を作る

成果物は 3 つ: **意図を持ってデザインされたと感じられる UI**、**ユーザーが安全に微調整できる構造**、
そして改善依頼では**見て分かる変化**。以下のフローを毎回守る。

## 絶対原則

1. **AI っぽさと物足りなさの両方を減点法で落とす。** `references/anti-ai-checklist.md` の
   ブラックリスト (過剰) と richness floor (不足) の両方に diff を突き合わせ、1 つでも該当したら
   直してから納品する。「無難で平凡」も AI っぽさの一形態。
2. **シンプルの基準は Apple HIG。** 装飾を足して豪華にするのではなく、階層・余白・タイポグラフィで
   豪華に見せ、リッチさはモーションと素材感で出す (`references/design-principles.md`)。
3. **聞くべきことは実装前に 1 回で聞く。** AskUserQuestion 1 回 (最大 4 問) でまとめて合意する。
4. **トークン駆動で作る。** 色・余白・角丸・フォント・モーションは必ず 1 箇所のトークン定義に
   集約する (`references/maintainability.md`)。
5. **モーションの既定はリッチ。** 動きのない UI は未完成として扱う (ミニマル合意時のみ例外)。
   物理は transform/opacity のみ + `prefers-reduced-motion` 必須 (`references/motion.md`)。
6. **アクセシビリティは非交渉。** コントラストは目視ではなく `scripts/validate-palette.mjs` で
   機械検証する。フォーカスリング・キーボード操作・セマンティック HTML を省かない。
7. **改善依頼は「変える」契約 (visible-delta contract)。** before/after 比較で視覚デルタが
   無ければ FAIL。既存デザインの尊重を理由に何も変えないのは失敗 — 保守が正しいのは拡張モードだけ。
8. **見ていない UI は納品しない。** スクリーンショットを撮り、自分の目 (Read) で確認するのが
   必須ゲート。撮影不能なときだけ「静的検証モード」を明示的に宣言してフォールバックする。

## モード判定 (最初に必ず)

| モード | 条件 | 基本姿勢 |
| --- | --- | --- |
| **新規** | UI がまだ無い / 独立した新ページ・新アプリ | ゼロから設計。プリセット + リッチ既定 |
| **拡張** | 既存デザインシステムのあるプロジェクトへの新画面・新部品の**追加** | 保守が正しい — 既存トークン・語彙の中で作り、新画面を浮かせない。勝手に新パレットを持ち込まない |
| **改善** | 既存の画面・UI への「改善して / リッチに / 見やすく / かっこよく / 今っぽく / Liquid Glass / 配色直して」 | **変えることが成果物** — 既存トークンは出発点であって制約ではない。`references/redesign-mode.md` を読んで Phase 0.5 へ |

- 判定軸は「対象が既に存在するか」×「依頼の動詞が品質向上か追加か」。**迷ったら改善に倒す**
  (改善モード Q1 の「トークン微調整」が保守側の意図を回収するため安全)。
- 拡張モード中に既存 UI の問題 (コントラスト不足等) を見つけたら、勝手に直さず報告で指摘する。

## 全体フロー

```text
共通     Phase 0    モード判定 + コンテキスト把握
改善のみ Phase 0.5  現状診断 (トークン採取・before スクショ・ギャップリスト)
共通     Phase 1    ボタン質問 1 回 (モード別・最大 4 問)
共通     Phase 2    デザイントークン定義 (ライト+ダーク) → 直後に validate-palette.mjs
共通     Phase 3    実装 (references のレシピを読み込んでから)
共通     Phase 4    検証ゲート 5 つ (全 PASS まで納品しない)
共通     Phase 5    報告 (改善時は before/after 要約) + 微調整ガイド
```

### Phase 0 — コンテキスト把握 (質問の前に必ず)

- 既存プロジェクト: デザイントークン / テーマ / Tailwind config、スタイリング手法の多数派、
  既存ページのトーン (2〜3 ファイル) を把握する。**この知識の使い方はモードで逆になる** —
  拡張では「従うため」、改善では「診断して変えるため」。
- 新規: 目的・対象ユーザー・デバイスを依頼文から読み取り、**情報設計ドラフト**
  (セクション構成・各セクションの要素) を作る。これが Phase 1 の質問の土台になる。

### Phase 0.5 — 現状診断 (改善モードのみ)

手順の詳細は `references/redesign-mode.md`。骨子:

1. **インベントリ採取**: スタック・トークン定義場所・現行実値 (色/フォント/影/角丸/モーション有無)
2. **before スクリーンショット**: 対象画面を 1440×900 で撮影し scratchpad に保存
   (撮影手段は Phase 4 と同じフォールバックチェーン。不能なら静的検証モードを宣言)
3. **ギャップ診断**: anti-ai-checklist の両方向 (過剰/不足) と突き合わせ、
   `| # | 症状 | 箇所 | 改善案 | 期待効果 |` のギャップリストを作る — Phase 1 の質問の根拠になる

### Phase 1 — AskUserQuestion で 1 回だけ質問 (最大 4 問)

共通ルール: 推測できるものは聞かず、推測値を推奨 (先頭ボタン) にして確認だけ取る。
質問の直前に、情報設計ドラフト (新規) / ギャップリスト要約 (改善) を**通常テキストで表示してから**聞く。
「お任せ」なら質問せず決定と理由を報告に含める。「その他」自由入力 (参考 URL 等) は最優先で反映する。

**新規モード**:

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | デザインの方向性 | 依頼に合う 3〜4 案 (例: ミニマル×余白 / エディトリアル / **ガラス&ディープ (Liquid Glass)** / ソフト&ウォーム)。各案に見出し書体・背景・レイアウトの言語スケッチ。アプリ UI・ダッシュボード・ダーク文脈ではガラス&ディープを必ず候補に入れ、ユーザーが Liquid Glass に言及済みならそれを推奨先頭に |
| 2 | カラーパレット | 下のプリセット早見表から方向性に合う 3 案。**ラベル=プリセット名、説明に `bg/text/accent` の実 hex + 印象 1 句**。ブランド色が既知なら「ブランド色 #… を軸に構成 (推奨)」を先頭に。紫グラデ系は出さない |
| 3 | 表示要素・構成 | 直前に表示した情報設計ドラフトへの確認ゲート: 「この構成で OK (推奨)」/「調整したい」 |
| 4 | 演出のリッチ度 | **「リッチ (推奨)」**: 主役演出 + スクロール出現 + 素材感 + 全対話要素のマイクロ / 「スタンダード」: ロード演出 + ホバーのみ / 「ミニマル」: フェードのみ (管理画面・ドキュメント向け)。説明に各レベルで何が動くかを書く |

**改善モード**:

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | 改善強度 | **「視覚リフレッシュ (推奨)」**: 構造は維持し色・タイポ・影・モーションを全面刷新 — before と並べて明確に違うことを保証 / 「全面リデザイン」: 情報設計・レイアウトから再設計 / 「トークン微調整」: 現行の雰囲気を保ち数カ所だけ (見た目の変化は小さい)。説明に診断ギャップ上位 2〜3 件を織り込み根拠を示す |
| 2 | 配色 | 「<プリセット名> に刷新 (推奨)」+ 実 hex / 対照的な第 2 プリセット / 「現行色を磨く」(現行 accent を保ちコントラスト・ランプ・配分だけ再構成)。ブランド色は無断で変えない |
| 3 | 演出のリッチ度 | 新規モードの Q4 と同一 3 択 (リッチ推奨)。説明に診断で見つかった不足 (「現状: hover なし・ロード演出なし」等) を書く |
| 4 | 適用範囲 | 「対象画面 + 共有基盤 (推奨)」: 共有トークン刷新、他画面の崩れも確認 / 「対象画面のみ」: スコープ付きトークンで局所化 / 「サイト全体」。単一画面プロジェクトなら省略して 3 問にする |

**拡張モード**: ①② は聞かず ③④ のみ (既存の語彙に従うため色・方向性の質問が不要)。

### Phase 2 — デザイントークン定義

**`references/color-system.md` と `references/maintainability.md` を読んでから**、合意内容を
1 箇所のトークン定義に落とす。以後の実装はトークンだけを参照する。

```css
:root {
  /* color: 選んだプリセットの Light 列を color-system.md §8 から転記 */
  --color-bg: …; --color-surface: …; --color-text: …; --color-text-secondary: …;
  --color-border: …; --color-accent: …; --color-on-accent: …; /* 明るい accent なら --color-accent-text も */

  /* typography: 見出しと本文でフォントを分ける (両方 Inter にしない) */
  --font-display: …; --font-body: …;

  /* spacing: 4/8px グリッド */
  --space-1: 0.5rem; --space-2: 1rem; --space-3: 1.5rem;
  --space-4: 2rem; --space-6: 3rem; --space-8: 4rem; --space-12: 6rem;

  /* shape & depth (影の 2 層実値は rich-techniques.md §2) */
  --radius-sm: 8px; --radius-md: 12px; --radius-lg: 16px;

  /* motion (スプリング実カーブは motion.md §2) */
  --duration-fast: 150ms; --duration-base: 250ms; --duration-slow: 450ms; --duration-spring: 500ms;
}
@media (prefers-color-scheme: dark) { :root { /* プリセットの Dark 列 */ } }
[data-theme="dark"] { /* 同上 — 手動切替・セクション局所反転用 (color-system.md §2) */ }
```

- **ライト・ダーク両テーマを初回から定義する** (プリセットが両方の検証済み値を持つ)
- Tailwind なら同内容を `@theme` / `theme.extend` に置く
- **定義直後に検証する** (手戻り防止):
  `node <スキルの場所>/scripts/validate-palette.mjs <tokens ファイル>`
  (インストール先に応じて `~/.claude/skills/ui-craft/scripts/` またはプロジェクトの
  `.claude/skills/ui-craft/scripts/`)。FAIL があればこの時点で直す
- 改善モード (視覚リフレッシュ以下) では**既存トークン名を保って値を刷新**する
  (redesign-mode.md §5 — 呼び出し側変更ゼロで全面が変わる)

### Phase 3 — 実装

**書き始める前に必ず読む**: `references/design-principles.md` と `references/motion.md`。
ミニマル合意時以外は `references/rich-techniques.md` も必ず。方向性がガラス系・Liquid Glass 言及時は
`references/liquid-glass.md` も必ず。

- レイアウトは情報の重要度から決める。**全要素等価のグリッドは思考放棄** — 1 画面の主役は 1 つ
- リッチ度の合意レベルに応じて motion.md §6 のマッピング表どおりレシピを組み込む。
  **rich-techniques.md 末尾の配合表 (必須下限) を満たすことが実装完了の条件**
- コンポーネントは「構造 (HTML/JSX)」「トークン」「振る舞い (JS)」を分離
- ダミーではなく**そのドメインらしい現実的なコンテンツ**を入れる (「Lorem ipsum」「Feature 1/2/3」禁止)
- 画像プレースホルダは灰色ボックスでなく自作図版 (rich-techniques.md §4)
- レスポンシブはモバイル幅 375px を実際に確認してから完了とする

### Phase 4 — 検証ゲート 5 つ (全 PASS まで納品しない)

1. **コントラスト機械検証**: `validate-palette.mjs` を最終トークンに実行 — **exit 0 必須**
2. **スクリーンショット**: desktop 1440×900 + mobile 375px を撮影し、**Read で開いて自分の目で確認**。
   モーションは 1 回リロードして初回ロードの見え方も確認。撮影手段のフォールバックチェーン:
   プロジェクトの playwright/puppeteer → `npx -y playwright screenshot --viewport-size=1440,900 <URL> <out.png>`
   (chromium 未取得なら `npx playwright install chromium`) → dev server は既存を `curl -sf` で優先し、
   無ければ run_in_background で起動して終了時に必ず止める → それも不可なら**静的検証モード**を宣言し、
   報告に「スクリーンショット検証未実施」とユーザー向け確認手順を明記
3. **アンチ AI チェック**: `references/anti-ai-checklist.md` の全項目 (richness floor 含む) と
   diff を突き合わせる。1 つでも該当 = 修正
4. **a11y / 挙動**: focus-visible・キーボード操作・`prefers-reduced-motion`・375px で崩れなし。
   「このデザインの意図を 1 文で言えるか?」— 言えなければ方向性から直す
5. **visible-delta 判定 (改善モードのみ)**: before/after を両方 Read で比較 (redesign-mode.md §7)。
   6 観点 (アクセント色 / 背景・サーフェス / 見出しタイポ / 深度言語 / モーション / 余白・レイアウト) 中、
   視覚リフレッシュ = **3 観点以上** + 並べて 1 秒で違いが分かる、全面リデザイン = 5 観点以上。
   FAIL → Phase 2 に戻って強化 (最大 2 周。届かなければ制約を添えてユーザーに確認)

### Phase 5 — 報告 + 微調整ガイド

報告には必ず含める (日本語・結論から):

1. 何を作った/変えたか + デザイン意図 1〜2 文 (方向性名・プリセット名を明示)
2. **改善モードのみ**: Before→After 要約表 (観点 × 1 行) + スクショ 2 枚の絶対パス +
   ギャップリスト消化状況 (直した件数・主要 3 件 / 意図的に残した項目と理由)
3. 検証結果: validate-palette PASS (ペア数・両テーマ)、visible-delta 判定、チェックリスト通過
4. **微調整ガイド**: 「色味 → `tokens.css` の `--color-accent`」のように、言いそうな修正 → 触る場所の
   対応表を実ファイルパス・実トークン名で。末尾に「色を変えたら validate-palette.mjs で再検証」を添える
5. 確認方法 (起動コマンド / ファイルパス)

### 納品後の修正リクエスト対応

- 「もう少し○○に」系 (色・速度・余白・角丸) → **トークンのみ変更** (`references/maintainability.md`)
- 要素の追加・削除 → 情報設計に影響するなら 1 問だけ確認、明確なら即実装
- **「改善して」「リッチに」「見やすく」系の品質向上依頼 → 改善モードとして再入する**
  (モード判定に戻る。「トークンのみ」ルールをこの種の依頼に適用してはならない)
- 方向性の変更 (「やっぱりダークで」等) → トークン一式の差し替えで対応できる構造になっているはず
