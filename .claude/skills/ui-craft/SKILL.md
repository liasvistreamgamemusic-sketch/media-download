# ui-craft — AI に見えない、Apple 級の UI を作る

このスキルの成果物は「見た目が良いコード」ではなく、**意図を持ってデザインされたと感じられる UI**と
**ユーザー自身が安全に微調整できる構造**の 2 つ。以下のフローを毎回守る。

## 絶対原則

1. **AI っぽさの排除は減点法でやる。** 実装後に `references/anti-ai-checklist.md` の
   ブラックリストと突き合わせ、1 つでも該当したら直してから納品する。
   「カードを 3 枚並べる」「紫グラデ」「絵文字アイコン」は無意識に出る — チェックで機械的に落とす。
2. **シンプルの基準は Apple HIG。** 装飾を足して豪華にするのではなく、階層 (hierarchy)・余白・
   タイポグラフィで豪華に見せる。リッチさはモーションと素材感で出す (`references/design-principles.md`)。
3. **聞くべきことは実装前に 1 回で聞く。** 色・トーン・表示要素・モーション量は
   AskUserQuestion でまとめて確認する。作ってから「イメージと違う」を防ぐのが最安。
4. **トークン駆動で作る。** 色・余白・角丸・フォント・モーションは必ず CSS variables
   (またはプロジェクトのテーマ機構) に集約する。「ちょっと青みを変えたい」が
   1 行の変更で済む構造にする (`references/maintainability.md`)。
5. **モーションは transform / opacity のみ。** `prefers-reduced-motion` を必ず尊重する。
   量ではなく質 — ロード時の 1 回のオーケストレーションが、散らばったマイクロアニメより効く
   (`references/motion.md`)。
6. **アクセシビリティは非交渉。** コントラスト比 AA (本文 4.5:1)、フォーカスリング、
   キーボード操作、セマンティック HTML。美しさの言い訳にしない。

## 全体フロー

```
Phase 0  コンテキスト把握 (既存資産・技術制約の確認)
Phase 1  情報設計ドラフト + ボタン質問 1 回 (トーン・色・要素・モーション量)
Phase 2  デザイントークン定義
Phase 3  実装 (references を読み込んでから)
Phase 4  アンチ AI チェックリストでセルフレビュー
Phase 5  報告 + 「どこを触れば何が変わるか」ガイド
```

### Phase 0 — コンテキスト把握 (質問の前に必ず)

既存プロジェクトの場合:

- 既存のデザイントークン / テーマ / Tailwind config / CSS variables を読む。
  **既にデザインシステムがあるなら、その語彙の中で作る** — 勝手に新パレットを持ち込まない。
- フレームワーク・スタイリング手法 (Tailwind / CSS Modules / styled-components / plain CSS) を特定し、
  プロジェクトの多数派に従う。
- 既存ページのトーンを 2〜3 ファイル見て把握する (新画面が浮かないように)。

新規の場合:

- 目的・対象ユーザー・デバイス (モバイル優先か) をユーザーの依頼文から読み取る。
- 読み取れた内容から**情報設計ドラフト** (セクション構成・各セクションに置く要素) を作っておく。
  これを Phase 1 の質問の直前に本文で提示する (質問 ③ の preview にも載せる)。

### Phase 1 — AskUserQuestion で 1 回だけ質問 (最大 4 問)

推測できるものは聞かない — 推測値を推奨ボタンにして確認だけ取る。
既存デザインシステムがある場合、①② は聞かずに ③④ だけ聞く。

**質問を出す直前に、情報設計ドラフト (質問 ③ の確認対象) を通常のテキストとして必ず表示する**。
preview はフォーカスした選択肢しか表示されない環境があるため、確認ゲートの内容を preview
だけに載せてはならない。比較用に preview を使う質問 (①④) も、要点 1 行は description に書く。

| # | 質問 | 選択肢の作り方 |
| --- | --- | --- |
| 1 | **デザインの方向性** | 依頼内容に合う方向性を 3〜4 案。各選択肢の preview に「見出しフォント・背景・レイアウトの言葉によるスケッチ」を書き、description に要点 1 行。例: ミニマル×余白 / エディトリアル (雑誌的) / ソフト&ウォーム / ダーク×プレシジョン。推奨を先頭に |
| 2 | **カラー** | ブランド色があるか不明なら聞く。提案する場合は「ベース (ほぼ無彩色) + アクセント 1 色」を具体的な色名+hex で 3 案。「紫グラデ」系は選択肢に入れない |
| 3 | **表示要素・構成** | Phase 0 で作った情報設計ドラフトを質問前に本文表示したうえで preview にも載せ、「この構成で OK (推奨)」/「要素を追加・削除したい」。ここが**要素があっているかの確認ゲート** |
| 4 | **モーション量** | 控えめ (フェードのみ) / 標準 (推奨: ロード時ステージング + ホバー) / リッチ (スクロール連動・視差含む)。preview に各レベルで何が動くかを書き、description に要点 1 行 |

- ユーザーが「お任せ」と言ったら質問せず、方向性の決定と理由を報告に含める。
- 回答の「その他」自由入力 (例: 参考サイト URL) は最優先で反映する。

### Phase 2 — デザイントークン定義

実装の最初に、合意した内容を **1 箇所のトークン定義**に落とす。以後の実装はトークンだけを参照する。

```css
:root {
  /* color: ベースはほぼ無彩色 + アクセント 1 色 (合意した hex) */
  --color-bg: #fafaf9;
  --color-surface: #ffffff;
  --color-text: #1c1917;
  --color-text-secondary: #78716c;
  --color-accent: /* Phase 1 で合意した色 */;
  --color-border: rgba(28, 25, 23, 0.08);

  /* typography: 見出しと本文でフォントを分ける (両方 Inter にしない) */
  --font-display: /* 個性のある見出し用 */;
  --font-body: /* 可読性の高い本文用 */;

  /* spacing: 4px または 8px グリッド */
  --space-1: 0.5rem; --space-2: 1rem; --space-3: 1.5rem;
  --space-4: 2rem;  --space-6: 3rem; --space-8: 4rem; --space-12: 6rem;

  /* shape & depth */
  --radius-sm: 6px; --radius-md: 10px; --radius-lg: 16px;
  --shadow-sm: 0 1px 2px rgba(0,0,0,0.04);
  --shadow-md: 0 4px 16px rgba(0,0,0,0.06);

  /* motion (references/motion.md の規定値。spring は @supports fallback 付きで) */
  --duration-fast: 150ms; --duration-base: 250ms; --duration-slow: 450ms;
  --ease-out: cubic-bezier(0.22, 1, 0.36, 1);
  --ease-snappy: cubic-bezier(0.16, 1, 0.3, 1);
}
```

Tailwind プロジェクトなら同じ内容を `theme.extend` / `@theme` に置く。値は例 — 合意内容で必ず差し替える。

### Phase 3 — 実装

**書き始める前に必ず読む**: `references/design-principles.md` (階層・タイポ・余白・色・素材) と
`references/motion.md` (easing・duration・オーケストレーション)。
リッチ寄りの依頼なら加えて `references/rich-techniques.md`。

実装ルール:

- レイアウトは情報の重要度から決める。**全要素等価のグリッドは思考放棄** —
  1 画面に主役は 1 つ、それを最大に、残りを従属させる。
- コンポーネントは「構造 (HTML/JSX)」「トークン (variables)」「振る舞い (JS)」を分離。
- ダミーデータではなく**そのドメインらしい現実的なコンテンツ**を入れる
  (「Lorem ipsum」「Feature 1/2/3」は AI っぽさの主因)。
- 画像プレースホルダは灰色ボックスではなく、CSS グラデーションメッシュ・図形・
  パターンなど文脈に合う代替を自作する。
- レスポンシブはモバイルを実際に確認してから完了とする (DevTools 幅 375px)。

### Phase 4 — セルフレビュー (納品前に毎回)

1. `references/anti-ai-checklist.md` を開き、全項目と diff を突き合わせる。該当があれば直す。
2. コントラスト比・フォーカス・キーボード操作・`prefers-reduced-motion` を確認。
3. ブラウザで実際に開いて確認する (スクリーンショット or 開発サーバ)。
   モーションは 1 回リロードして初回ロードの見え方を体感する。
4. 「このデザインの意図を 1 文で言えるか?」— 言えなければ方向性がブレている。直す。

### Phase 5 — 報告 + 微調整ガイド

報告には必ず以下を含める (日本語・結論から):

1. 何を作ったか + デザイン意図 1〜2 文
2. **微調整ガイド**: 「色味を変える → `tokens.css` の `--color-accent` 1 行」
   「余白を詰める → `--space-*`」「アニメ速度 → `--duration-*`」のように、
   ユーザーが言いそうな修正 → 触るべき場所の対応表
3. 確認方法 (起動コマンド / ファイルパス)

### 納品後の修正リクエスト対応

- 「もう少し○○に」系 (色・速度・余白・角丸) → **トークンのみ変更**。構造を触らない。
- 要素の追加・削除 → 情報設計に影響するなら 1 問だけ確認、明確なら即実装。
- 方向性の変更 (「やっぱりダークで」等) → トークン一式の差し替えで対応できる構造に
  なっているはず。なっていなければそれは Phase 2 の失敗 — 構造ごと直す。
