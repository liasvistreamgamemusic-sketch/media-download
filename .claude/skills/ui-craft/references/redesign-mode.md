# 改善モード (redesign) — 「変える」を契約にする手順書

改善モードと判定したら Phase 0.5 の前に必ず読む。
このモードの成果物は**見て分かる変化**。既存デザインの尊重を理由に何も変えないのは失敗であり、
before/after 比較で視覚デルタが無ければ Phase 4 で FAIL になる (visible-delta contract)。

## 1. モード判定の詳細

| シグナル | 判定 |
| --- | --- |
| 「改善して」「リッチに」「見やすく」「かっこよく」「今っぽく」「Liquid Glass にして」「配色を直して」× 既存画面 | **改善** |
| 「〜画面を追加」「〜コンポーネントを作って」× 既存デザインシステム | 拡張 (保守が正しい — SKILL.md Phase 0 へ) |
| 対象 UI がまだ存在しない | 新規 |

- **迷ったら改善に倒す**。安全な理由: 改善モードの Q1 に「トークン微調整」があり、保守側の意図は
  そこで回収できる。逆 (拡張に倒す) は「頼んだのに変わらない」の再発経路
- 納品後の「もう少し○○に」は maintainability.md の管轄 (トークン微調整)。ただし納品後でも
  「改善して/リッチに」系の品質向上依頼が来たら、このモードに再入する

## 2. 診断手順 (Phase 0.5 — 質問より前)

読む順序: ①トークン定義 (CSS variables / Tailwind config / テーマファイル) → ②対象画面の実装 →
③共有レイアウト・共通コンポーネント。読んでいない UI を診断しない。

**トークン採取表** (会話内に作る — Phase 1 の質問と Phase 2 の刷新対象になる):

| 項目 | 現状値 | 所感 |
| --- | --- | --- |
| スタック / スタイリング手法 | 例: plain CSS 単一ファイル | |
| bg / surface / text / accent | 実 hex | 彩度・コントラストの目視所感 |
| フォント / サイズスケール | | 階層の強弱 |
| 影 / 角丸 / ボーダー | | 深度言語の有無 |
| モーション | @keyframes 数・transition 箇所 | hover/focus/ロード演出の有無 |
| ダークモード | 対応有無 | |

**before スクリーンショット**: dev server を検出 (`curl -sf`) または起動し、対象画面を 1440×900 で撮影。
保存先はセッションの scratchpad 配下 `ui-craft/before-<画面名>.png` (**リポジトリ内に置かない**)。
撮影手段のフォールバックチェーンは SKILL.md Phase 4 と同じ。撮影不能なら「静的検証モード」を宣言して続行し、
Phase 4 の比較をコード検査で代替する (§7)。

## 3. ギャップ診断

anti-ai-checklist.md の**両方向** — ブラックリスト (過剰) と richness floor (不足) — に現状 UI を突き合わせ、
ギャップリストを作る:

| # | 症状 | 箇所 (file:line) | 改善案 | 期待効果 |
| --- | --- | --- | --- | --- |
| 1 | 例: hover が背景色変化のみ・focus リング無し | app.css:120 | lift+影+focus-visible (motion.md R5) | 触感・a11y |
| 2 | 例: アクセントが極小タグのみ、CTA が黒 | tokens.css:8 | ブランドアンカー刷新 + CTA に accent | 主役の明確化 |

- 診断は「良くする余地」の列挙であって断罪ではない — 既存の意図 (密度優先など) を汲んだ改善案にする
- リスト上位 2–3 件は Phase 1 の質問 (改善強度・配色・リッチ度) の description に織り込み、
  「これを直すには視覚リフレッシュ以上が必要」という根拠として提示する

## 4. 改善強度別の作業範囲

| 強度 | 触ってよい | 触らない | 納品条件 |
| --- | --- | --- | --- |
| トークン微調整 | トークン値のみ | 構造・クラス・演出の追加 | 合意した変更点の反映 |
| **視覚リフレッシュ (推奨)** | クラス / スタイル / トークン / 演出 / アセット全面 | **DOM 構造・コンポーネント境界・ルーティング・状態管理** | visible-delta 6 観点中 **3 以上** |
| 全面リデザイン | 情報設計・レイアウトから再構築 | 機能・データフロー | 6 観点中 **5 以上** + レイアウト変化 |

- 視覚リフレッシュの「構造維持」は**機能を壊さない**ための線引き。セレクタや class 名の変更は可、
  JS が参照する id / data 属性・イベント配線は維持する
- 全面リデザインは以降、新規モードの Phase 2–3 に合流する (診断ギャップリストを情報設計の入力にする)
- どの強度でも**機能を壊さないことが絶対条件** — 変更後に対象画面の主要操作を一通り動かして確認する

## 5. 既存トークンとの付き合い方

原則: **既存トークンは出発点であって制約ではない** (拡張モードの「語彙の中で作る」はここでは適用しない)。

1. 既存トークンが役割名 (`--color-bg` 等) → **名前を保って値を刷新**する。呼び出し側変更ゼロで全面が変わる
2. 既存トークンが値名 (`--blue-500`, `--gray-3`) → 役割名のエイリアス層を足し、コンポーネント側は
   段階的にエイリアスを参照させる (`--color-accent: var(--blue-500)` から始めて値を差し替える)
3. トークンが無い (色がハードコード) → Phase 2 でトークン層を新設し、対象範囲の実値を置換する。
   これ自体が改善の主要成果物になる
4. ブランド色 (ロゴ・ユーザーが名指しした色) は無断で変えない — 「現行色を磨く」(コントラスト・ランプ・
   配分だけ OKLCH で再構成) を質問の選択肢に必ず入れる

## 6. 波及管理 (適用範囲の回答別)

- **対象画面 + 共有基盤 (推奨)**: 共有トークンを更新し、影響する他画面を grep で列挙 →
  主要画面をスクショまたは目視で崩れ確認。確認した画面名を報告に書く
- **対象画面のみ**: トークンをスコープ化して局所適用 (`.screen-x { --color-accent: … }` または
  data 属性スコープ)。グローバルトークンは触らない
- **サイト全体**: 画面リストを作り、共有トークン刷新 → 画面ごとに崩れ確認・調整の順で回る (工数最大)

## 7. before/after 比較 (Phase 4 ゲート 5)

1. before (Phase 0.5) と after のスクショを**両方 Read で開いて**見比べる
2. 6 観点で変化を数える: ①アクセント色 ②背景/サーフェス ③見出しタイポ (書体 or スケール)
   ④深度言語 (影/ボーダー/ガラス) ⑤モーション (新規演出の追加) ⑥余白リズム/レイアウト密度
3. 判定: 視覚リフレッシュ = **3 観点以上の明確な変化** + 「2 枚を並べて 1 秒で違いが分かる」。
   全面リデザイン = 5 観点以上 + レイアウト構造の変化。トークン微調整 = 合意点の反映のみ確認 (delta 契約対象外)
4. **FAIL したら**: どの観点が動いていないかを特定し、Phase 2 に戻って該当トークン/演出を強める。
   再検証は最大 2 周 — 2 周で届かなければ、何が制約になったか (ブランド色固定・構造制約等) を添えて
   ユーザーに指示を仰ぐ (無限ループ防止)
5. 静的検証モード時の代替判定: 変更された色トークン ≥ 5 個、またはコード上で 3 観点以上の変更を diff で確認
6. 報告 (Phase 5) には Before→After 要約表・スクショ 2 枚の絶対パス・ギャップリスト消化状況
   (直した件数 + 主要 3 件 / 意図的に残した項目 + 理由) を含める
